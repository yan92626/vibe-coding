## 1. 分层控制策略

掌握分层控制是提升 Vibe Coding 效率的关键。不是所有事情都让 AI 做，也不是所有事情都自己做，而是**各司其职，发挥各自优势**。

### 1.1 什么该你控制

#### 层次 1：战略层（必须你控制）

**架构设计**
```
你决定：
- 整体架构（前后端分离？单体应用？）
- 技术栈选择（React？Vue？）
- 模块划分（如何组织代码？）
- 数据流设计（状态如何管理？）

为什么必须你控制：
- 影响整个项目
- 难以后期修改
- 需要全局视角
- AI 缺少业务理解
```

**核心业务逻辑**
```
你决定：
- 业务规则（VIP 用户如何计算折扣？）
- 数据模型（用户、订单如何关联？）
- 关键算法（推荐算法如何实现？）
- 安全策略（权限如何控制？）

为什么必须你控制：
- 需要深入理解业务
- 涉及核心价值
- 错误影响大
- AI 可能理解错
```

**质量标准**
```
你决定：
- 什么是"足够好"
- 哪些地方必须完美
- 哪些地方可以妥协
- 何时重构

为什么必须你控制：
- 只有你知道优先级
- 只有你知道约束条件
- AI 不能做价值判断
```

#### 层次 2：战术层（你主导，AI 辅助）

**接口设计**
```
你设计：
- API 接口（RESTful 还是 GraphQL？）
- 函数签名（参数和返回值）
- 组件接口（props 定义）
- 数据结构（对象结构）

AI 辅助：
- 提供最佳实践建议
- 生成接口文档
- 实现接口代码
```

**错误处理**
```
你设计：
- 错误处理策略
- 用户提示方式
- 日志记录方案
- 降级方案

AI 辅助：
- 实现 try-catch
- 生成错误提示
- 添加日志代码
```

**性能优化**
```
你决定：
- 哪里需要优化
- 优化到什么程度
- 优化的优先级

AI 辅助：
- 提供优化建议
- 实现优化代码
- 性能测试
```

### 1.2 什么该 AI 生成

#### 层次 1：执行层（AI 主导）

**样板代码**
```
完全让 AI 生成：
- CRUD 操作
- 表单验证
- 数据格式化
- 工具函数

原因：
- 重复性高
- 模式固定
- 出错风险低
- AI 擅长
```

**UI 组件**
```
让 AI 生成：
- 基础组件（Button、Input）
- 布局组件（Header、Footer）
- 展示组件（Card、List）

你审查：
- 样式是否符合要求
- 交互是否合理
- 可访问性
```

**测试代码**
```
让 AI 生成：
- 单元测试
- 集成测试
- 测试数据

你审查：
- 测试覆盖率
- 测试场景是否完整
- 边界情况
```

**文档和注释**
```
让 AI 生成：
- 函数注释
- API 文档
- README
- 使用说明

你审查：
- 准确性
- 完整性
- 可读性
```

#### 层次 2：辅助层（AI 辅助，你决策）

**代码重构**
```
AI 提供：
- 重构建议
- 重构后的代码
- 对比说明

你决定：
- 是否采纳
- 何时重构
- 重构范围
```

**Bug 修复**
```
AI 提供：
- 问题分析
- 修复方案
- 修复代码

你决定：
- 方案是否合理
- 是否有副作用
- 是否需要测试
```

### 1.3 如何划分边界

#### 决策框架

```
问自己这些问题：

1. 这个决策的影响范围？
   - 全局 → 你控制
   - 局部 → AI 可以

2. 这个决策的可逆性？
   - 难逆转 → 你控制
   - 易逆转 → AI 可以

3. 这个决策需要的知识？
   - 业务知识 → 你控制
   - 技术知识 → AI 可以

4. 出错的后果？
   - 严重 → 你控制
   - 轻微 → AI 可以

5. 你的理解程度？
   - 完全理解 → AI 可以
   - 不太理解 → 你控制或学习后再决定
```

#### 实际案例

**案例 1：用户认证系统**

```
你控制：
认证策略（Session？JWT？）
密码策略（长度、复杂度）
权限模型（RBAC？ABAC？）
安全要求（加密、防攻击）

AI 生成：
登录表单 UI
密码加密代码
Session 管理代码
权限检查函数

协作：
你："用 JWT 认证，密码用 bcrypt 加密，
    至少 8 位包含数字和字母"
AI：生成实现代码
你：审查安全性，测试
```

**案例 2：数据展示页面**

```
你控制：
展示什么数据
数据如何组织
交互逻辑
性能要求

AI 生成：
表格组件
图表组件
分页逻辑
排序筛选

协作：
你："展示用户列表，包含姓名、邮箱、注册时间，
    支持分页和搜索"
AI：生成表格和搜索功能
你：调整样式和交互
```

### 1.4 动态调整边界

边界不是固定的，随着你的成长而变化：

#### 初学者阶段

```
你控制：
- 核心需求（要做什么）
- 基本判断（对不对）

AI 生成：
- 大部分代码
- 技术方案
- 实现细节

特点：
- AI 做得多
- 你学得多
- 逐步理解
```

#### 熟练阶段

```
你控制：
- 架构设计
- 核心逻辑
- 关键决策
- 代码审查

AI 生成：
- 样板代码
- 辅助功能
- 测试代码
- 文档

特点：
- 分工明确
- 效率最高
- 质量可控
```

#### 专家阶段

```
你控制：
- 所有关键决策
- 复杂逻辑
- 性能优化
- 架构演进

AI 生成：
- 重复性工作
- 标准化代码
- 辅助工具

特点：
- 你主导一切
- AI 是助手
- 精确控制
```

## 2. 约束引导技术

给 AI 明确的约束，能大幅提升产出质量。约束不是限制，而是**精确表达你的意图**。

### 2.1 历史视角：Prompt Engineering 的实践演进

**2024年上半年的探索**

在将 AI 应用到数据加工生产环节时，我系统实践了 prompt engineering 的各种技术：

**Zero-shot 的痛点**：
- 对措辞极其敏感，换个词结果完全不同
- 同样的 prompt 在不同数据上表现差异巨大
- 需要大量测试才能找到"有效"的表达

**Few-shot 的挑战**：
- Examples 的选择至关重要
- 选不好的 examples 反而降低效果
- 需要针对不同数据类型准备不同的 examples
- Examples 太多会占用大量 token

**CoT（思维链）的问题**：
- 推理链条容易跑偏，尤其是复杂逻辑
- Token 消耗大，成本高
- 需要精心设计推理步骤

**ReAct（推理-行动）的复杂性**：
- Action 和 Observation 的设计需要反复调试
- 循环次数难以控制
- 容易陷入死循环

**当时缺少的**：
- 系统化的方法论来指导这些技术的组合使用
- 没有工具帮助管理和测试不同的 prompt 版本
- 缺乏对"为什么有效"的深入理解
- 无法量化评估不同 prompt 的效果

**现代工具的改进**：

**Claude Code 的 Skill 系统 - 具体例子**：

2024 年我做数据清洗时的 prompt：
```
"帮我清洗这个 CSV 文件：
1. 删除空行
2. 统一日期格式为 YYYY-MM-DD
3. 去除重复记录
4. 处理缺失值...
[每次都要重复这一大段]"
```

现在有了 Skill：
```
/data-cleaning

[Skill 自动注入]：
- 项目的数据格式规范
- 常见的数据问题处理方式
- 输出格式要求
- 错误处理策略

我只需要说："清洗 users.csv"
```

更强大的是，Skill 会根据文件类型自动调整：
- CSV 文件：关注格式、编码、分隔符
- JSON 文件：关注结构、嵌套、类型
- 数据库导出：关注 NULL 值、外键、索引

**Cursor 的 .cursorrules - 实际场景**：

我的项目有个规则："所有 API 接口必须有错误处理和日志记录"

没有 .cursorrules 时：
```
我："写一个获取用户信息的接口"
AI：[生成代码，但没有错误处理]
我："加上错误处理"
AI：[加上 try-catch]
我："加上日志记录"
AI：[加上 console.log]
我："日志要用 winston，不是 console.log"
AI：[修改]
```

有了 .cursorrules：
```
// .cursorrules
所有 API 接口必须：
- 使用 try-catch 包裹
- 用 winston 记录日志
- 返回统一的错误格式
- 添加请求 ID 用于追踪

我："写一个获取用户信息的接口"
AI：[直接生成符合所有规则的代码]
```

**更强的模型 - 真实对比**：

**2023 年用 GPT-3.5**：
```
我："写一个函数，验证邮箱格式"
AI：生成一个简单的正则，但有漏洞
我："要考虑国际化邮箱"
AI：修改正则，但还是有问题
我："要处理 + 号和点号"
AI：再修改...
[来回 5 轮才对]
```

**2026 年用 Claude Opus**：
```
我："写一个函数，验证邮箱格式"
AI：[直接生成考虑了各种边界情况的代码]
   - 国际化域名
   - 特殊字符
   - 长度限制
   - RFC 5322 标准
[一次就对]
```

**但更重要的认知转变**：

**从"研究技术"到"清晰沟通"**

2024 年我会想：
- "这个场景用 Few-shot 还是 CoT？"
- "Examples 应该选哪几个？"
- "推理步骤怎么设计？"

2026 年我会想：
- "我到底要什么？"
- "有哪些约束条件？"
- "什么是'好'的标准？"

具体例子：

**技术思维**：
```
"用 Few-shot 方式，给 3 个例子：
例子 1：输入 A，输出 B
例子 2：输入 C，输出 D
例子 3：输入 E，输出 F
现在处理：输入 G"
```

**沟通思维**：
```
"我要处理用户输入，要求：
- 去除首尾空格
- 转换为小写
- 只保留字母和数字
- 长度限制 3-20 字符

不要：
- 使用正则表达式（团队不熟悉）
- 抛出异常（返回错误信息）

示例：
输入 '  Hello123!  ' → 输出 'hello123'
输入 'ab' → 输出错误 '长度不足'"
```

第二种方式：
- 不需要研究"用什么技术"
- 直接说清楚"要什么、不要什么"
- 给一个例子就够了（不是 Few-shot 的 3 个例子，而是"说明预期"的例子）

**从"Examples"到"约束"**

**2024 年的做法**：
```
"参考这些代码：
[粘贴 10 个不用 jQuery 的例子]
现在写一个不用 jQuery 的轮播图"
```

**2026 年的做法**：
```
"写一个轮播图，要求：
- 不使用 jQuery（项目已移除）
- 用原生 JS 或 React
- 支持触摸滑动
- 自动播放可配置"
```

为什么第二种更好？
- 不需要准备 10 个例子
- 约束更明确（"不用 jQuery"直接说，不用让 AI 从例子中推断）
- AI 可以用最新的最佳实践（而不是模仿可能过时的例子）

**从"CoT 推理步骤"到"提供上下文"**

**2024 年的做法**：
```
"分析这个 bug：
第 1 步：检查输入参数
第 2 步：检查函数调用
第 3 步：检查返回值
第 4 步：检查错误处理
..."
```

**2026 年的做法**：
```
"这个函数报错了：[错误信息]

背景：
- 这是支付流程的一部分
- 昨天刚上线，之前测试都正常
- 只有部分用户遇到（约 5%）
- 错误日志显示：[日志]

帮我分析可能的原因"
```

为什么第二种更好？
- 不需要设计"推理步骤"（AI 自己会推理）
- 提供的是"有用的上下文"（背景信息）
- AI 可以根据实际情况调整分析路径

**对 Vibe Coding 的启示**：

**技术会过时，沟通能力不会**

Zero-shot、Few-shot、CoT 这些技术，随着模型变强，可能不再需要。但"如何清晰表达意图"这个能力，永远重要。

具体来说：
- 2023 年：需要 Few-shot 是因为模型不够聪明
- 2026 年：模型够聪明了，直接说清楚就行
- 但"说清楚"本身是个能力：知道说什么、不说什么、怎么说

**约束比示例更有效**

给 10 个例子，不如说清楚 3 个约束：
- 例子：AI 要从中推断规律，可能推断错
- 约束：直接告诉 AI 规则，不会误解

**上下文比技巧更重要**

与其研究"怎么问"，不如提供"有用的信息"：
- 项目背景
- 技术约束
- 业务规则
- 预期效果

这些信息，比任何 prompt 技巧都有用。

### 2.2 如何给出有效约束

#### 约束类型 1：技术约束

**明确技术栈**
```
模糊：
"帮我做一个网站"

明确：
"用 React + Vite + Tailwind CSS 做一个网站"

更明确：
"用 React 18 + Vite 4 + Tailwind CSS 3，
不使用 TypeScript，
不使用状态管理库，
用 React Hooks"
```

**明确技术限制**
```
"要求：
- 兼容 Chrome、Firefox、Safari 最新版
- 不需要兼容 IE
- 移动端优先
- 不使用 jQuery
- 使用 ES6+ 语法"
```

**明确性能要求**
```
"性能要求：
- 首屏加载 < 2 秒
- 列表渲染 1000 条数据不卡顿
- 图片懒加载
- 代码分割"
```

#### 约束类型 2：代码风格约束

**命名规范**
```
"代码规范：
- 变量用驼峰命名（camelCase）
- 常量用大写下划线（UPPER_SNAKE_CASE）
- 组件用帕斯卡命名（PascalCase）
- 文件名用短横线（kebab-case）"
```

**代码组织**
```
"代码组织：
- 每个函数不超过 30 行
- 每个文件不超过 200 行
- 相关功能放在一起
- 导入语句按类型分组"
```

**注释要求**
```
"注释要求：
- 每个函数添加 JSDoc 注释
- 复杂逻辑添加行内注释
- 不要注释显而易见的代码
- 用中文注释"
```

#### 约束类型 3：功能约束

**明确功能范围**
```
"功能要求：
必须有：
- 用户登录
- 任务列表
- 添加任务

不需要：
- 用户注册（管理员添加）
- 任务分类（保持简单）
- 任务提醒（以后再说）"
```

**明确交互方式**
```
"交互要求：
- 点击任务切换完成状态（不要用复选框）
- 双击任务进入编辑模式
- 按 Enter 保存，按 Esc 取消
- 删除前要确认"
```

**明确数据处理**
```
"数据要求：
- 用 localStorage 存储（不用后端）
- 数据格式：JSON
- 每次操作后自动保存
- 页面加载时自动读取"
```

#### 约束类型 4：质量约束

**错误处理**
```
"错误处理：
- 所有异步操作用 try-catch
- 用户友好的错误提示
- 错误记录到 console
- 不要让应用崩溃"
```

**边界情况**
```
"边界情况：
- 空列表显示提示信息
- 输入为空不能提交
- 网络错误显示重试按钮
- 数据加载显示 loading"
```

**可访问性**
```
"可访问性：
- 所有按钮有 aria-label
- 表单有 label
- 键盘可操作
- 颜色对比度符合 WCAG AA"
```

### 2.3 约束的层次（技术栈、风格、架构）

#### 层次 1：架构约束（最重要）

```
"架构要求：
- 前后端分离
- RESTful API
- 组件化开发
- 单向数据流
- 模块化设计"

影响：整个项目
时机：项目开始时确定
可变性：难以改变
```

#### 层次 2：技术栈约束（重要）

```
"技术栈：
前端：
- React 18
- React Router 6
- Axios
- Tailwind CSS

后端：
- Node.js + Express
- MongoDB
- JWT 认证"

影响：开发方式
时机：项目开始时确定
可变性：可以改变，但成本高
```

#### 层次 3：代码风格约束（中等）

```
"代码风格：
- 遵循 Airbnb 规范
- 使用 Prettier 格式化
- 使用 ESLint 检查
- 函数式编程优先"

影响：代码质量
时机：随时可以确定
可变性：容易改变
```

#### 层次 4：实现细节约束（灵活）

```
"实现细节：
- 使用 useState 管理状态
- 使用 useEffect 处理副作用
- 使用 async/await 处理异步
- 使用箭头函数"

影响：具体实现
时机：实现时确定
可变性：很容易改变
```

### 2.4 示例对比（模糊 vs 精确）

#### 示例 1：登录功能

**模糊约束**
```
"帮我做一个登录功能"

AI 可能做出：
- 复杂的 OAuth 登录
- 包含注册、找回密码
- 使用复杂的状态管理
- 过度设计
```

**精确约束**
```
"实现用户登录功能，要求：

技术：
- React Hooks（useState）
- Axios 发送请求
- JWT Token 存储在 localStorage

功能：
- 只要登录，不要注册
- 用户名 + 密码
- 记住登录状态
- 登录失败显示错误

UI：
- 简洁的表单
- 居中布局
- 响应式设计

错误处理：
- 网络错误提示
- 登录失败提示
- 输入验证"

AI 会生成：

## 3. 迭代优化而非重写

掌握增量修改的艺术，避免"推倒重来"的陷阱。

### 3.1 增量修改的艺术

#### 原则 1：小步快跑

```
大步修改：
"重写整个登录模块，
改用 Redux 管理状态，
重新设计 UI，
添加第三方登录"

问题：
- 改动太大
- 风险高
- 难以回退
- 不知道哪里出错

小步修改：
第 1 步："添加错误提示"
→ 测试，OK

第 2 步："改进 UI 样式"
→ 测试，OK

第 3 步："添加记住密码"
→ 测试，OK

优势：
- 每步可控
- 风险低
- 容易回退
- 问题易定位
```

#### 原则 2：保持功能不变

```
重构时：
改进代码结构
优化性能
提升可读性

但是：
不改变功能
不添加新功能
不删除功能

原因：
- 重构和功能分开
- 降低风险
- 容易验证
```

#### 原则 3：每次修改后测试

```
修改流程：
1. 确定要改什么
2. 修改代码
3. 立即测试
4. 确认 OK
5. 提交版本
6. 再改下一个

不要：
1. 改很多地方
2. 一起测试
3. 出错了不知道哪里错
```

### 3.2 如何描述修改意图

#### 方法 1：明确修改范围

```
模糊：
"优化一下代码"

明确：
"只优化 handleSubmit 函数，
提取重复的验证逻辑，
其他代码不动"
```

#### 方法 2：说明修改原因

```
"修改 login 函数，
原因：当前没有错误处理，
      网络错误时应用崩溃

修改内容：
- 添加 try-catch
- 添加错误提示
- 添加重试按钮"
```

#### 方法 3：提供修改前后对比

```
"当前行为：
点击按钮 → 直接提交 → 可能失败

期望行为：
点击按钮 → 验证输入 → 提交 → 错误处理

修改：
在提交前添加验证逻辑"
```

#### 方法 4：使用"只改 X"模式

```
"只改 CSS，不改 HTML 和 JS"
"只改错误处理，不改业务逻辑"
"只改命名，不改功能"
"只添加注释，不改代码"
```

### 3.3 迭代优化的实践

#### 实践 1：功能优先，质量其次

```
第一版：实现功能
→ 能用就行
→ 不追求完美
→ 快速验证

第二版：优化质量
→ 重构代码
→ 改进命名
→ 添加注释

第三版：性能优化
→ 优化算法
→ 减少渲染
→ 缓存数据

第四版：完善细节
→ 边界情况
→ 错误处理
→ 用户体验
```

#### 实践 2：问题驱动优化

```
不要：
- 没问题也优化
- 为了优化而优化
- 过度设计

而是：
- 发现问题再优化
- 解决实际问题
- 够用就好

例子：
发现：列表滚动卡顿
→ 分析：数据量大，全量渲染
→ 优化：虚拟滚动
→ 验证：不卡了
→ 完成
```

#### 实践 3：渐进式重构

```
不要：
"这个文件太乱了，全部重写"

而是：
第 1 步：提取重复代码
第 2 步：改进命名
第 3 步：拆分大函数
第 4 步：优化结构
第 5 步：添加注释

每一步都：
- 保持功能不变
- 立即测试
- 确认 OK
```

#### 实践 4：A/B 对比

```
当不确定优化是否更好时：

1. 保留原版本（版本 A）
2. 创建优化版本（版本 B）
3. 对比两个版本
4. 选择更好的
5. 删除另一个

不要：
- 直接覆盖原版本
- 优化后发现不如原来
- 无法回退
```

### 3.4 避免重写的技巧

#### 技巧 1：识别重写信号

```
这些情况可能需要重写：
代码完全无法理解
修改成本 > 重写成本
技术栈完全过时
架构根本性问题

但大多数情况：
可以渐进式重构
不需要推倒重来
增量优化更安全
```

#### 技巧 2：重构而非重写

```
重构：
- 改进代码结构
- 保持功能不变
- 小步迭代
- 风险可控

重写：
- 从零开始
- 可能改变功能
- 一次性完成
- 风险高

优先选择重构
```

#### 技巧 3：部分重写

```
如果必须重写：

不要：
- 重写整个应用

而是：
- 重写最需要的模块
- 其他模块保持不变
- 逐步替换

例子：
重写登录模块
→ 测试，OK
→ 其他模块不动
→ 再重写下一个模块
```

#### 技巧 4：新旧并存

```
重写策略：

1. 创建新版本
2. 新旧版本并存
3. 逐步迁移功能
4. 验证新版本
5. 删除旧版本

优势：
- 可以随时回退
- 风险低
- 用户无感知
```

长对话容易丢失上下文，掌握对话管理技巧很重要。

### 6.1 保持上下文连贯

#### 技巧 1：定期总结

```
每隔一段时间（如 10-15 轮对话）：

你："我们总结一下当前状态：

项目：任务管理应用

技术栈：
- React + Vite
- Tailwind CSS
- localStorage

已完成：
- 任务列表展示
- 添加任务
- 标记完成

当前正在做：
- 任务编辑功能

下一步：
- 任务删除
- 数据持久化优化

请确认你理解了这些信息"

AI："确认理解"
```

#### 技巧 2：重要信息重复

```
重要决策要反复强调：

第 1 次："我们决定用 localStorage，不用后端"
第 5 次："记住，我们用 localStorage 存储"
第 10 次："继续用 localStorage，不要建议后端"

关键约束要经常提醒：
"保持简单，不要过度设计"
"只实现核心功能"
"代码要容易理解"
```

#### 技巧 3：使用代码注释

```
在代码中记录决策：

```javascript
// 技术决策：
// 1. 使用 localStorage（不用后端）
//    原因：个人项目，数据量小
// 2. 不使用 Redux（用 useState）
//    原因：状态简单，不需要复杂管理
// 3. 使用 Tailwind CSS
//    原因：快速开发，不需要写 CSS

function App() {
  // ...
}
```

每次 AI 看到代码，就会看到这些决策
```

#### 技巧 4：创建项目文档

```
创建 PROJECT.md：

# 任务管理应用

## 技术决策
- React + Vite
- Tailwind CSS
- localStorage（不用后端）
- 不用 Redux

## 功能范围
必须有：
- 添加任务
- 完成任务
- 删除任务

不需要：
- 用户系统
- 任务分类
- 提醒功能

## 开发原则
- 保持简单
- 快速迭代
- 够用就好

每次新对话，先让 AI 读这个文档
```

### 6.2 何时开启新对话

#### 场景 1：对话太长

```
信号：
- 超过 50 轮对话
- AI 开始忘记早期信息
- 响应变慢

行动：
1. 总结当前状态
2. 保存重要信息
3. 开启新对话
4. 提供上下文
```

#### 场景 2：切换任务

```
信号：
- 要做完全不同的功能
- 切换到不同的模块
- 开始新的阶段

行动：
1. 完成当前任务
2. 提交代码
3. 开启新对话
4. 说明新任务
```

#### 场景 3：方向调整

```
信号：
- 发现方向错了
- 需要重新开始
- 技术方案改变

行动：
1. 总结经验教训
2. 明确新方向
3. 开启新对话
4. 避免旧信息干扰
```

#### 场景 4：AI 表现异常

```
信号：
- AI 理解错误
- 产出质量下降
- 重复犯错

行动：
1. 开启新对话
2. 重新提供上下文
3. 更清晰的描述
```

### 6.3 新对话的开场

#### 开场模板

```
新对话开场：

"我正在开发 [项目名称]

项目背景：
[简要说明项目是什么]

技术栈：
[列出主要技术]

当前状态：
[已完成什么，正在做什么]

本次任务：
[这次对话要做什么]

重要约束：
[关键的技术决策和约束]

请确认你理解了这些信息，
然后我们开始"
```

#### 开场示例

```
"我正在开发一个任务管理应用

项目背景：
个人项目，帮助我管理每天的任务

技术栈：
- React 18 + Vite
- Tailwind CSS
- localStorage（不用后端）

当前状态：
- 已完成：任务列表、添加、完成
- 正在做：任务编辑功能

本次任务：
实现任务编辑功能，
双击任务进入编辑模式

重要约束：
- 保持简单，不过度设计
- 用 useState 管理状态（不用 Redux）
- 代码要容易理解

请确认你理解了这些信息"
```
