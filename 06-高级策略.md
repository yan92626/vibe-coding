## 1. 知识差距弥补法

这是我们在对话开始时深入讨论的核心策略：如何在不懂技术的情况下，仍然能高效使用 Vibe Coding。

### 1.1 用 AI 学习技术概念

#### 学习策略 1：概念优先，细节其次

```
不要：
"教我 React 的所有知识"
→ 太宽泛，学不完

而是：
"我想用 React 做任务管理，
需要理解哪些核心概念？"
→ 聚焦，够用就好
```

**AI 回答示例**：
```
核心概念：
1. 组件（Component）- 构建 UI 的积木
2. 状态（State）- 数据的存储
3. Props - 组件间传递数据
4. 事件处理 - 响应用户操作

对于任务管理，主要用到：
- useState（管理任务列表）
- 事件处理（添加、删除任务）
- 列表渲染（显示任务）

先理解这些就够了
```

#### 学习策略 2：用类比理解

```
你："什么是 React 的 state？"

AI："State 就像一个记事本：
- 你在上面写东西（设置状态）
- 内容变了，页面就更新（重新渲染）
- 每个组件有自己的记事本（独立状态）

例子：
任务列表就是一个 state
添加任务 = 在记事本上写一条
删除任务 = 从记事本上划掉一条"

你：理解了！
```

#### 学习策略 3：边做边学

```
第 1 步：看最简单的例子
你："给我一个最简单的 React 例子"
AI：生成 Hello World
你：运行，理解基本结构

第 2 步：加一点功能
你："加一个按钮，点击显示文字"
AI：添加按钮和事件
你：运行，理解事件处理

第 3 步：加状态管理
你："让按钮点击次数显示出来"
AI：添加 useState
你：运行，理解状态

第 4 步：应用到实际项目
你："现在我理解了，帮我做任务管理"
```

### 1.2 两步法：先学习，再实现

这是核心方法，我们之前详细讨论过：

#### 第一步：AI 当技术顾问（不写代码）

```
你："我想做 [功能]，但我不懂技术。

请告诉我：
1. 需要考虑哪些关键问题？
2. 有哪些常见方案？
3. 每个方案的优缺点？
4. 对于我的场景，推荐哪种？
5. 为什么推荐这个？"

AI：详细解释各种方案

你：理解后，做出决策
```

#### 第二步：基于理解后再要代码

```
你："好，我决定用 [方案]。

现在请实现，要求：
- [具体约束]
- [关键要求]
- [注意事项]"

AI：生成代码

你：因为理解了方案，能判断对错
```

#### 两步法的优势

```
你理解了技术方案
能判断 AI 的输出
遇到问题知道怎么办
学到了知识

vs 一步法（直接要代码）：
不理解代码
无法判断对错
出问题不知道怎么办
什么都没学到
```

### 1.3 如何快速建立技术直觉

#### 方法 1：多问"为什么"

```
不要只接受答案，要理解原因：

AI："用 useState 管理状态"
你："为什么用 useState？"
AI："因为状态变化时需要重新渲染"
你："什么时候不用 useState？"
AI："数据不需要触发渲染时"
你："明白了"

通过追问，建立深入理解
```

#### 方法 2：对比学习

```
你："useState 和普通变量有什么区别？"

AI："对比：
普通变量：
- 改变不会触发渲染
- 组件重新渲染时会重置
- 适合临时数据

useState：
- 改变会触发渲染
- 数据在渲染间保持
- 适合 UI 相关数据"

你：通过对比，理解差异
```

#### 方法 3：实验验证

```
你："我想验证一下 useState 的行为"

AI：生成实验代码

你：运行，观察
- 改变 state → 页面更新 ✓
- 改变普通变量 → 页面不更新 ✓

你：通过实验，建立直觉
```

#### 方法 4：总结模式

```
做了几个项目后，总结模式：

"我发现的模式：
1. 需要显示在页面上的数据 → 用 state
2. 临时计算的数据 → 用普通变量
3. 不变的配置 → 用常量
4. 需要在组件间共享 → 用 props 或 context

这是我的直觉"

通过总结，形成自己的理解
```

## 2. 复杂项目的 Vibe Coding

当项目变大变复杂时，需要更系统的策略。

### 2.1 大型项目的分解策略

#### 策略 1：垂直切片

```
不要：按技术层分解
第 1 周：做所有的数据库
第 2 周：做所有的 API
第 3 周：做所有的 UI

而是：按功能切片
第 1 周：用户登录（数据库+API+UI）
第 2 周：任务管理（数据库+API+UI）
第 3 周：数据统计（数据库+API+UI）

优势：
- 每周都有可用的功能
- 及早发现问题
- 可以调整优先级
```

#### 策略 2：MVP 优先

```
第一版：最小可用产品
- 只做核心功能
- 能用就行
- 快速验证

第二版：完善功能
- 添加重要功能
- 优化体验
- 修复问题

第三版：打磨产品
- 添加辅助功能
- 性能优化
- 完善细节

不要一开始就追求完美
```

#### 策略 3：模块化开发

```
项目结构：
project/
├── auth/          # 认证模块（独立）
├── tasks/         # 任务模块（独立）
├── users/         # 用户模块（独立）
└── shared/        # 共享代码

每个模块：
- 独立开发
- 独立测试
- 清晰接口
- 低耦合

优势：
- 可以并行开发
- 容易维护
- 容易测试
```

#### 策略 4：迭代深化

```
第 1 轮：实现基础版本
- 核心流程能跑
- 不考虑边界情况
- 不考虑性能

第 2 轮：完善功能
- 处理边界情况
- 添加错误处理
- 改进用户体验

第 3 轮：优化提升
- 性能优化
- 代码重构
- 完善测试

每一轮都是完整可用的
```

### 2.2 架构层面的 Vibe Coding

#### 架构设计流程

```
第 1 步：和 AI 讨论架构
你："我要做一个 [项目描述]，
    预计有 [功能列表]，
    请建议整体架构"

AI：提供架构方案

你：理解、评估、调整

第 2 步：确定技术栈
你："基于这个架构，
    推荐什么技术栈？"

AI：提供技术选型建议

你：根据团队情况决策

第 3 步：设计模块划分
你："如何划分模块？
    模块间如何通信？"

AI：提供模块设计

你：审查、调整

第 4 步：定义接口
你："定义各模块的接口"

AI：生成接口定义

你：审查、确认

第 5 步：逐个实现模块
用 Vibe Coding 实现每个模块
```

#### 架构决策记录

```
创建 ARCHITECTURE.md：

# 架构设计

## 整体架构
[架构图或描述]

## 技术栈
前端：React + Vite + Tailwind
后端：Node.js + Express
数据库：PostgreSQL
缓存：Redis

## 模块划分
- auth：用户认证
- api：业务 API
- web：前端应用
- shared：共享代码

## 关键决策
1. 为什么用 PostgreSQL？
   - 需要关系型数据
   - 需要事务支持
   
2. 为什么用 Redis？
   - Session 存储
   - 缓存热点数据

## 数据流
[数据流图或描述]

## 接口设计
[API 接口文档]
```

### 2.3 团队协作中的应用

#### 协作模式 1：个人 Vibe + 团队审查

```
个人开发：
- 用 Vibe Coding 快速实现
- 自己审查代码
- 本地测试

团队审查：
- 提交 PR
- 团队 Code Review
- 讨论和改进
- 合并代码

优势：
- 个人效率高
- 团队质量控制
- 知识共享
```

#### 协作模式 2：结对 Vibe Coding

```
两人协作：
- A：和 AI 对话，生成代码
- B：审查代码，提出问题
- 共同决策
- 实时讨论

优势：
- 实时审查
- 减少错误
- 知识共享
- 决策更好
```

#### 协作模式 3：分工协作

```
架构师：
- 设计整体架构
- 定义接口
- 技术选型

开发者：
- 用 Vibe Coding 实现模块
- 遵循架构设计
- 实现接口

优势：
- 架构统一
- 开发高效
- 质量可控
```

#### 团队规范

```
建立团队规范：

1. 代码规范
   - 统一的代码风格
   - 统一的命名规范
   - 统一的项目结构

2. AI 使用规范
   - 哪些可以用 AI
   - 哪些必须人工
   - 如何审查 AI 代码

3. 提交规范
   - 提交信息格式
   - PR 描述要求
   - 审查标准

4. 文档规范
   - 技术决策记录
   - API 文档
   - 代码注释
```

## 3. 不同 AI 工具的组合使用

不同 AI 工具有不同优势，组合使用能发挥最大价值。

### 3.1 Claude、ChatGPT、Copilot 的优势场景

#### Claude 的优势

```
最适合：
复杂的技术讨论
需要深入理解的场景
学习新技术
架构设计讨论
代码审查和优化建议

特点：
- 理解能力强
- 解释清晰
- 适合对话式学习
- 代码质量高

使用场景：
"我想理解 React 的渲染机制"
"帮我设计这个系统的架构"
"审查这段代码，给出改进建议"
```

#### ChatGPT 的优势

```
最适合：
快速查询
简单问题
代码片段生成
文档编写
多语言支持

特点：
- 响应快
- 知识面广
- 插件生态丰富
- 多模态能力

使用场景：
"这个错误是什么意思？"
"写一个正则表达式匹配邮箱"
"生成 API 文档"
```

#### GitHub Copilot 的优势

```
最适合：
代码补全
重复性代码
标准模式代码
日常编码
IDE 内使用

特点：
- 无缝集成
- 实时补全
- 上下文理解好
- 不打断流程

使用场景：
- 写函数时自动补全
- 生成测试用例
- 重复性代码
```

### 3.2 工具切换策略

#### 策略 1：按阶段切换

```
探索阶段：Claude
- 讨论技术方案
- 学习新概念
- 架构设计

实现阶段：Copilot
- 日常编码
- 代码补全
- 快速实现

问题解决：ChatGPT
- 快速查询错误
- 查找解决方案
- 搜索文档

审查阶段：Claude
- 代码审查
- 优化建议
- 重构方案
```

#### 策略 2：按任务类型切换

```
复杂任务：Claude
- 需要深入讨论
- 需要理解业务
- 需要设计方案

简单任务：Copilot
- 标准化代码
- 重复性工作
- 快速补全

查询任务：ChatGPT
- 快速查询
- 简单问题
- 文档查找
```

#### 策略 3：组合使用

```
例子：实现新功能

第 1 步：用 Claude 讨论方案
"我想实现 [功能]，
有哪些方案？
推荐哪个？"

第 2 步：用 Copilot 实现
在 IDE 中编码
Copilot 自动补全

第 3 步：遇到问题用 ChatGPT
"这个错误怎么解决？"
快速查询

第 4 步：用 Claude 审查
"审查这段代码，
有什么问题？"

结果：发挥各工具优势
```

### 3.3 工具选择决策树

```
开始
├─ 需要深入讨论吗？
│  ├─ 是 → Claude
│  └─ 否 → 继续
├─ 在 IDE 中编码吗？
│  ├─ 是 → Copilot
│  └─ 否 → 继续
├─ 需要快速查询吗？
│  ├─ 是 → ChatGPT
│  └─ 否 → 继续
└─ 需要代码审查吗？
   ├─ 是 → Claude
   └─ 否 → 根据具体情况
```

## 4. 元认知与自我优化

最高级的策略是**持续改进自己的 Vibe Coding 能力**。

### 4.1 记录和分析自己的 Vibe Coding 过程

#### 记录什么

```
每次 Vibe Coding 后记录：

1. 任务描述
   - 要做什么
   - 预期时间
   - 实际时间

2. 使用的方法
   - 用了哪些技巧
   - 哪些有效
   - 哪些无效

3. 遇到的问题
   - 什么问题
   - 如何解决
   - 花了多久

4. 产出质量
   - 代码质量如何
   - 是否符合预期
   - 需要返工吗

5. 经验教训
   - 学到了什么
   - 下次如何改进
```

#### 记录方式

```
方式 1：简单日志
2024-01-15
任务：实现用户登录
时间：2 小时（预期 1 小时）
方法：两步法（先讨论方案，再实现）
问题：密码加密不熟悉，花了 30 分钟学习
质量：良好，代码清晰
教训：下次先学习不熟悉的技术

方式 2：结构化记录
创建 LEARNING.md：
# Vibe Coding 学习日志

## 2024-01-15 用户登录
- 任务：[描述]
- 方法：[使用的技巧]
- 问题：[遇到的问题]
- 解决：[如何解决]
- 教训：[经验教训]

方式 3：使用工具
- Notion
- Obsidian
- 简单的 Markdown 文件
```

#### 定期回顾

```
每周回顾：
□ 这周做了什么？
□ 哪些做得好？
□ 哪些需要改进？
□ 学到了什么？

每月回顾：
□ 这个月的进步？
□ 效率提升了吗？
□ 代码质量如何？
□ 新掌握的技巧？

每季度回顾：
□ 整体能力提升？
□ 还有哪些不足？
□ 下季度目标？
```

### 4.2 识别个人模式

#### 识别有效模式

```
通过记录，识别你的有效模式：

"我发现的模式：
1. 用两步法（先讨论再实现）效率最高
2. 小步迭代比大步修改更可控
3. 早上精力好，适合做复杂任务
4. 下午适合做重复性工作
5. 每 1 小时休息一次效率更高"

这些是你的个人模式
```

#### 识别无效模式

```
识别你的反模式：

"我发现的问题：
1. 直接要代码（不讨论方案）经常返工
2. 一次改太多地方容易出错
3. 不测试就继续经常积累问题
4. 晚上写代码质量差
5. 连续工作超过 2 小时效率下降"

避免这些反模式
```

#### 识别个人偏好

```
了解你的偏好：

"我的偏好：
1. 喜欢用 Claude（解释清晰）
2. 喜欢简单的技术栈
3. 喜欢看到快速反馈
4. 不喜欢过度设计
5. 重视代码可读性"

根据偏好调整工作方式
```

### 4.3 持续改进

#### 改进循环

```
PDCA 循环：

Plan（计划）：
- 设定改进目标
- 制定改进计划

Do（执行）：
- 尝试新方法
- 记录过程

Check（检查）：
- 评估效果
- 对比之前

Act（行动）：
- 保留有效的
- 调整无效的
- 继续改进
```

#### 改进方向

```
方向 1：提升效率
- 减少返工
- 提高准确性
- 加快速度

方向 2：提升质量
- 代码更清晰
- 更少 bug
- 更好的设计

方向 3：扩展能力
- 学习新技术
- 掌握新工具
- 尝试新方法

方向 4：优化流程
- 简化步骤
- 自动化重复工作
- 建立模板
```

#### 实践建议

```
建议 1：设定小目标
不要：一次改进所有
而是：每次改进一个方面

例子：
本周目标：提升需求描述能力
- 使用需求模板
- 每次都明确约束
- 记录效果

建议 2：实验验证
不确定某个方法是否有效？
- 尝试一周
- 记录数据
- 对比效果
- 决定是否采用

建议 3：向他人学习
- 看优秀项目的代码
- 学习他人的方法
- 参考最佳实践
- 吸收有用的

建议 4：分享和教学
- 写博客记录经验
- 教别人 Vibe Coding
- 在教学中深化理解
- 获得反馈改进
```

#### 成长路径

```
初学者（0-3 个月）：
- 掌握基本工作流
- 能完成简单项目
- 建立基本直觉

熟练者（3-12 个月）：
- 掌握进阶技巧
- 能处理复杂项目
- 有自己的模式库

专家（12 个月+）：
- 精通各种策略
- 能指导他人
- 持续创新方法
```
