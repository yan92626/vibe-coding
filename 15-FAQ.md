## 基础问题

### Q1: Vibe Coding 适合我吗？

**适合你，如果**：
- ✅ 想快速实现想法
- ✅ 愿意尝试新方法
- ✅ 接受"不完美"的过程
- ✅ 想提升开发效率
- ✅ 愿意边做边学

**需要调整期望，如果**：
- ⚠️ 完全零基础 - 可以用 Vibe Coding 学习，但要从简单项目开始，不要期望立即做出生产级系统
- ⚠️ 追求完美主义 - 需要接受"足够好"的理念
- ⚠️ 项目风险极高 - 建议结合传统方法，增加审查环节
- ⚠️ 有严格的合规要求 - 需要额外的质量控制流程

**特别说明 - 零基础也可以开始**：
- 💡 完全不懂编程的人也可以使用 Vibe Coding
- 💡 从简单的 demo 和小项目开始是很好的学习路径
- 💡 通过与 AI 对话，可以边做边学习编程概念
- 💡 关键是要理解：学习阶段 ≠ 生产阶段
  - 学习阶段：做 demo、小项目、实验性项目 ✅
  - 生产阶段：需要积累一定基础后再考虑 ⏳

**零基础学习路径**：
```
第 1 阶段（1-2周）：
- 通过 AI 学习基本概念（变量、函数、循环）
- 做非常简单的 demo（计算器、待办事项）
- 重点：理解而非记忆

第 2 阶段（1-2月）：
- 做稍微复杂的小项目
- 学习调试和问题解决
- 重点：建立编程思维

第 3 阶段（3-6月）：
- 做完整的个人项目
- 学习最佳实践
- 重点：质量和可维护性

第 4 阶段（6月+）：
- 考虑生产级项目
- 深入学习技术栈
- 重点：专业化发展
```

**建议**：
- 零基础：从 demo 开始，把 Vibe Coding 作为学习工具 ✅
- 有基础：可以直接用于实际项目 ✅
- 任何阶段：都要保持学习和理解，不要只是复制粘贴 ⚠️

### Q2: 我需要什么编程基础才能开始？

**这取决于你的目标**：

**如果目标是学习编程（做 demo、小项目）**：
- ✅ 零基础也可以开始
- ✅ 通过与 AI 对话边做边学
- ✅ 从最简单的项目开始
- ✅ 重点是理解而非记忆

**如果目标是做实际项目**：
- 建议先具备：
  - 理解基本编程概念（变量、函数、循环）
  - 能看懂简单代码
  - 会使用命令行基础操作
  - 能判断代码"看起来对不对"

**如果目标是生产级系统**：
- 需要：
  - 扎实的编程基础
  - 技术栈经验
  - 调试和问题解决能力
  - 代码质量意识

**不需要**：
- ❌ 精通某种语言
- ❌ 记住所有 API
- ❌ 深入理解底层原理
- ❌ 有多年经验

**推荐路径**：
```
零基础 → 学习基础概念（通过 Vibe Coding）
       → 做简单项目（1-2月）
       → 做完整项目（3-6月）
       → 考虑生产级项目（6月+）
```

### Q3: Vibe Coding 会让我变得不会编程吗？

**不会，原因**：

1. **你仍然需要判断** - 判断 AI 的输出对不对，判断方案是否合适
2. **你会学到更多** - 从 AI 学习新技术，理解不同的实现方式
3. **改变的是方式，不是能力** - 从"手写代码"到"设计和判断"

**类比**：用计算器不会让你失去数学能力，用 AI 不会让你失去思考能力。

**关键**：保持主动学习，理解 AI 生成的代码。

### Q4: 哪个 AI 工具最适合 Vibe Coding？

**没有"最好"，只有"最适合"**：

- **学习新技术**：Claude（解释清晰）
- **日常开发**：Cursor 或 Copilot（IDE 集成）
- **快速查询**：ChatGPT（响应快）
- **代码审查**：Claude（分析深入）

**推荐组合**：
- 预算充足：Cursor + Claude
- 中等预算：Claude Pro
- 低预算：Claude 免费版 + Codeium

详见：[[11-工具选择指南]]

### Q5: Vibe Coding 和传统编程可以结合吗？

**可以，而且应该结合**：

**混合策略**：
- 核心业务逻辑 → 传统方法（精心设计）
- 辅助功能 → Vibe Coding（快速实现）
- 原型验证 → Vibe Coding（快速试错）
- 生产代码 → 传统方法审查 + Vibe Coding 实现

**不是非此即彼**，而是"灵活组合，发挥优势"。

## 实践问题

### Q6: 如何判断 AI 生成的代码是否可靠？

**判断方法**：

1. **运行测试** - 运行代码，测试功能，看是否符合预期
2. **代码审查** - 检查逻辑是否正确，有没有明显错误
3. **理解原理** - 问自己：这段代码在做什么？为什么这样写？
4. **查阅文档** - 关键部分查看官方文档，验证 API 用法
5. **咨询 AI** - 让 AI 审查代码，指出潜在问题

**原则**：信任但验证，关键代码必须理解，不理解的不要用。

---

### Q7: AI 总是理解错我的需求怎么办？

**解决方法**：

1. **提供更多上下文** - 详细描述需求，包括背景、目标、约束
2. **使用示例** - 提供参考网站、截图或类似功能
3. **确认理解** - 让 AI 先描述理解，再确认或纠正
4. **迭代澄清** - 多轮对话逐步明确需求
5. **换个说法** - 用不同角度和词汇描述

详见：[[08-常见问题与解决方案#A2]]

---

### Q8: 如何避免代码越改越乱？

**预防方法**：

1. **小步迭代** - 每次只改一个地方，测试后再继续
2. **使用版本控制** - 每次改动 git commit，出问题可以回退
3. **明确改动目标** - 改动前明确要改什么、为什么改
4. **定期重构** - 发现代码乱了先重构整理，再继续开发
5. **设置检查点** - 每完成一个功能停下来审查整理

详见：[[08-常见问题与解决方案#D1]]

### Q9: 什么时候应该自己写代码而不是用 AI？

**自己写的场景**：

1. **核心业务逻辑** - 需要深入理解，错误影响大
2. **你完全理解的代码** - 自己写更快，不需要 AI
3. **学习新概念** - 手写加深理解，建立肌肉记忆
4. **高风险代码** - 安全相关、性能关键、需要精确控制

**决策原则**：问自己这段代码重要吗？我理解吗？风险高吗？AI 能做好吗？

详见：[[08-常见问题与解决方案#E1]]

---

### Q10: 如何处理 AI 生成的安全漏洞？

**处理方法**：

1. **明确安全要求** - 在需求中说明密码加密、防注入、防 XSS 等
2. **安全审查** - 让 AI 审查代码的安全性，指出风险
3. **使用安全库** - 使用成熟的库（bcrypt、helmet、validator）
4. **代码审查** - 关键代码自己审查、同事审查、安全专家审查
5. **安全测试** - 渗透测试、安全扫描、漏洞检测

**原则**：安全相关代码必须仔细审查，不理解的安全代码不要用。

详见：[[08-常见问题与解决方案#C3]]

## 效率问题

### Q11: 为什么我的 Vibe Coding 效率很低？

**可能原因**：

1. **需求描述不清** - AI 理解错，反复返工 → 使用需求模板，明确约束
2. **没有验证就继续** - 问题积累，难以定位 → 每一步都测试验证
3. **追求完美** - 陷入细节，浪费时间 → "足够好"就继续
4. **不会使用工具** - 工具用不好，效率低 → 学习工具使用技巧
5. **没有建立模式库** - 每次从零开始 → 积累常用模式

**诊断方法**：记录一次完整的 Vibe Coding 过程，分析哪里花时间最多，针对性改进。

详见：[[10-疑难杂症诊断]]

---

### Q12: 如何加快迭代速度？

**方法**：

1. **减小迭代粒度** - 每次做一点，不要一次做很多
2. **快速验证** - 写完立即运行，不要等到"完成"
3. **使用模板** - 常用功能用模板，不要每次从零开始
4. **并行工作** - 等待 AI 时思考下一步，充分利用时间
5. **减少返工** - 需求明确再开始，每步验证再继续

---

### Q13: 如何减少返工？

**方法**：

1. **需求明确** - 开始前确认要做什么、不做什么、约束条件
2. **方案讨论** - 实现前讨论有哪些方案、各有什么优劣
3. **小步验证** - 每一步做一点、测试、确认对、再继续
4. **及时调整** - 发现不对立即停止、分析原因、调整方向
5. **记录决策** - 重要决策记录为什么这样做，避免重复讨论

### Q14: 多个 AI 工具如何配合使用？

**配合策略**：

**按阶段分工**：
- 探索阶段：Claude（讨论方案）
- 实现阶段：Cursor/Copilot（编码）
- 调试阶段：ChatGPT（快速查询）
- 审查阶段：Claude（代码审查）

**按任务分工**：
- 复杂任务：Claude（深入讨论）
- 简单任务：Copilot（快速补全）
- 查询任务：ChatGPT（快速查询）

**实践建议**：不要频繁切换，每个工具用在最擅长的地方。

详见：[[06-高级策略#3]]

---

### Q15: 如何管理长对话的上下文？

**管理方法**：

1. **定期总结** - 每 10-15 轮对话总结项目状态、技术栈、进展
2. **重要信息重复** - 关键决策反复强调
3. **代码注释记录** - 在代码中记录技术决策和原因
4. **创建项目文档** - PROJECT.md 记录技术决策、功能范围、开发原则
5. **知道何时开启新对话** - 超过 50 轮、AI 开始忘记、切换任务时

详见：[[05-进阶技巧#6]]

## 进阶问题

### Q16: Vibe Coding 能用于大型项目吗？

**可以，但需要策略**：

**适用方式**：
- 架构设计：传统方法（详细规划）
- 模块实现：Vibe Coding（快速实现）
- 代码审查：传统方法（质量保证）

**关键策略**：模块化设计、清晰的接口、严格的审查、完善的测试

**注意事项**：不能完全依赖 AI，需要人工把控方向，代码质量要严格审查。

详见：[[06-高级策略#2]]

### Q17: 团队如何采用 Vibe Coding？

**采用步骤**：

1. **试点项目** - 选择低风险项目、小团队、短周期验证效果
2. **建立规范** - 制定代码规范、AI 使用规范、审查标准
3. **培训团队** - 培训 Vibe Coding 理念、工具使用、最佳实践
4. **逐步推广** - 个人使用 → 小团队 → 全团队 → 标准化
5. **持续优化** - 收集反馈、改进流程、分享经验

### Q18: 如何在 Vibe Coding 中保证代码质量？

**质量保证方法**：

1. **明确质量标准** - 在需求中说明代码规范、命名规范、注释要求
2. **代码审查** - 自己审查、AI 审查、同事审查
3. **测试验证** - 单元测试、集成测试、手动测试
4. **迭代优化** - 第 1 版能跑、第 2 版优化、第 3 版完善
5. **使用工具** - Linter、Formatter、静态分析

### Q19: Vibe Coding 的局限性是什么？

**局限性**：

1. **不适合所有场景**
   - 关键业务逻辑（需要深入理解和精确控制）
   - 安全敏感代码（需要专业安全审查）
   - 高性能要求（需要精细优化）
   - 强合规要求（需要严格的质量控制）

2. **需要判断力（但可以培养）**
   - 学习阶段：通过实践逐步建立判断力 ✅
   - 实际项目：需要能判断 AI 输出的基本正确性
   - 生产环境：需要扎实的技术判断力
   - **关键**：判断力是可以通过 Vibe Coding 过程培养的

3. **AI 的固有局限**
   - 可能理解错需求
   - 可能生成错误代码
   - 不理解具体业务逻辑
   - 需要人工验证和审查

4. **质量保证挑战**
   - 需要人工审查（尤其是关键代码）
   - 可能遗漏边界情况
   - 测试覆盖需要补充
   - 安全性需要额外关注

5. **团队协作挑战**
   - 代码风格可能不一致
   - 知识传递需要文档化
   - 需要建立团队规范
   - 需要统一的质量标准

**不同阶段的应对**：

**学习阶段**：
- 局限性影响较小
- 重点是学习和理解
- 可以接受不完美
- 通过实践建立判断力

**实际项目阶段**：
- 需要更多验证
- 建立质量检查流程
- 结合传统方法
- 持续学习提升

**生产环境阶段**：
- 严格的质量控制
- 完善的测试覆盖
- 专业的代码审查
- 安全和性能保证

**应对方法**：
- 了解局限性，不盲目使用
- 在合适场景使用
- 结合传统方法
- 持续学习和改进
- 建立质量保证机制

### Q20: Vibe Coding 的未来发展方向？

**趋势预测**：

1. **AI 能力提升** - 更强的理解能力、更好的代码质量、更少的错误
2. **工具集成** - IDE 深度集成、多工具协同、无缝工作流
3. **个性化** - 学习个人风格、适应团队规范、定制化建议
4. **协作增强** - 团队协作支持、知识共享、代码审查辅助
5. **新的范式** - 从"写代码"到"设计系统"、从"实现"到"创造"

**对开发者的影响**：技能要求变化、工作方式改变、价值创造转移、持续学习重要。
