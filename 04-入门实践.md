## 1. 准备工作

在开始 Vibe Coding 之前，做好这些准备能让你事半功倍。

### 1.1 选择合适的 AI 工具

#### 主流工具对比

**Claude (Code/Web)**
- ✅ 对话能力强，理解复杂需求
- ✅ 代码质量高
- ✅ 擅长解释和教学
- ⚠️ 有使用限制（免费版）
- 💡 **推荐用于**：学习新技术、复杂逻辑、需要详细解释

**ChatGPT**
- ✅ 响应速度快
- ✅ 知识面广
- ✅ 插件生态丰富
- ⚠️ 代码质量不如 Claude
- 💡 **推荐用于**：快速查询、简单任务、多语言支持

**GitHub Copilot**
- ✅ IDE 集成，无缝体验
- ✅ 代码补全快速
- ✅ 上下文理解好
- ⚠️ 对话能力弱
- 💡 **推荐用于**：日常编码、代码补全、重复性工作

**Cursor**
- ✅ 专为编程设计的 IDE
- ✅ 集成 AI 对话和编辑
- ✅ 项目上下文理解好
- ⚠️ 需要付费
- 💡 **推荐用于**：完整项目开发、团队协作

**Codex (通过 API)**
- ✅ 可定制化
- ✅ 可集成到自己的工具
- ⚠️ 需要编程能力
- 💡 **推荐用于**：高级用户、自定义工具

#### 新手推荐

**如果你是完全新手**：
- 首选：**Claude** (claude.ai)
- 理由：对话友好，解释清晰，适合学习

**如果你有编程基础**：
- 首选：**Claude Code** 或 **Cursor**
- 理由：IDE 集成，效率更高

**如果你预算有限**：
- 首选：**Claude 免费版** + **GitHub Copilot 试用**
- 理由：免费或低成本，功能够用

#### 工具组合建议

**最佳组合**：
```
主力工具：Claude Code / Cursor（日常开发）
辅助工具：Claude Web（学习和探索）
补充工具：GitHub Copilot（代码补全）
```

### 1.3 心态准备

#### 心态 1：接受"不完美"

**传统思维**：
> "我要先把所有东西都学会，再开始写代码"

**Vibe Coding 思维**：
> "我先做出来，不懂的边做边学"

**实践建议**：
- ✅ 允许第一版不完美
- ✅ 相信迭代的力量
- ✅ 不要追求一次到位

#### 心态 2：相信但验证

**错误心态**：
> "AI 说什么就是什么，我全盘接受"

**正确心态**：
> "AI 的建议很有价值，但我要验证"

**实践建议**：
- ✅ 相信 AI 的能力
- ✅ 但要验证关键输出
- ✅ 培养自己的判断力

#### 心态 3：保持好奇心

**被动心态**：
> "AI 帮我做完就行，我不需要理解"

**主动心态**：
> "AI 为什么这样做？我能学到什么？"

**实践建议**：
- ✅ 问"为什么"
- ✅ 理解 AI 的建议
- ✅ 从 AI 那里学习

#### 心态 4：快速试错

**恐惧心态**：
> "我怕做错，所以要想清楚再做"

**实验心态**：
> "试试看，错了再改，成本很低"

**实践建议**：
- ✅ 降低试错成本
- ✅ 快速验证想法
- ✅ 从错误中学习

#### 心态 5：保持主导

**依赖心态**：
> "AI 帮我决定吧，我不知道怎么做"

**主导心态**：
> "我知道要什么，AI 帮我实现"

**实践建议**：
- ✅ 你是项目的主人
- ✅ AI 是你的助手
- ✅ 关键决策你来做

## 2. 第一个 Vibe Coding 项目

让我们通过一个完整的实例，体验 Vibe Coding 的完整流程。

### 2.1 选择合适的起点项目

#### 好的起点项目特征

✅ **范围小**：1-2 天能完成
✅ **需求清晰**：知道要做什么
✅ **有反馈**：能看到运行结果
✅ **有价值**：对你有实际用处

#### 推荐的起点项目

**项目 1：个人任务管理器**
- 功能：添加、完成、删除任务
- 技术：HTML + JavaScript 或 React
- 时间：2-4 小时
- 学习点：基础 CRUD、状态管理

**项目 2：天气查询工具**
- 功能：输入城市，显示天气
- 技术：调用天气 API
- 时间：1-2 小时
- 学习点：API 调用、数据展示

**项目 3：Markdown 笔记应用**
- 功能：编辑和预览 Markdown
- 技术：React + Markdown 库
- 时间：2-3 小时
- 学习点：第三方库使用、实时预览

**项目 4：个人博客生成器**
- 功能：从 Markdown 生成静态博客
- 技术：Node.js + 静态站点生成
- 时间：3-4 小时
- 学习点：文件处理、模板引擎

#### 我们选择：任务管理器

接下来，我们用任务管理器作为示例，完整演示 Vibe Coding 流程。

### 2.2 完整的实践示例（对话流程）

#### 阶段 1：理解需求（5 分钟）

**你 → AI**：
```
我想做一个简单的任务管理器，但我不太确定具体要做什么功能。

请帮我理解：
1. 一个基础的任务管理器通常包含哪些功能？
2. 对于初学者，应该先实现哪些核心功能？
3. 用什么技术栈比较简单？
```

**AI → 你**：
```
一个基础任务管理器通常包含：

核心功能：
1. 添加任务
2. 标记任务完成/未完成
3. 删除任务
4. 显示任务列表

可选功能：
5. 编辑任务
6. 任务分类
7. 任务优先级
8. 数据持久化

建议先实现核心功能 1-4。

技术栈建议：
- 最简单：纯 HTML + JavaScript（无需构建工具）
- 现代化：React + Vite（学习现代框架）
- 后端：暂时不需要，用 localStorage 存储

推荐：从纯 HTML + JavaScript 开始，能跑起来后再考虑升级。
```

**你的思考**：
- ✅ 明白了基础功能
- ✅ 决定先做核心功能
- ✅ 选择纯 HTML + JavaScript（最简单）

#### 阶段 2：技术方案（10 分钟）

**你 → AI**：
```
好的，我决定用纯 HTML + JavaScript。

请告诉我：
1. 需要哪些文件？
2. 大概的代码结构是什么样的？
3. 有什么需要注意的地方？

先不要写代码，让我理解整体思路。
```

**AI → 你**：
```
文件结构：
```
project/
├── index.html    # 页面结构
├── style.css     # 样式
└── app.js        # 逻辑代码
```

代码结构：

index.html：
- 输入框（添加任务）
- 按钮（提交）
- 任务列表容器

app.js：
- 任务数组（存储任务）
- 添加任务函数
- 删除任务函数
- 切换完成状态函数
- 渲染列表函数
- 从 localStorage 加载/保存

注意事项：
1. 用 localStorage 持久化数据
2. 每次操作后重新渲染列表
3. 给每个任务一个唯一 ID
```

**你的思考**：
- ✅ 结构很清晰
- ✅ 理解了基本思路
- ✅ 准备开始实现

#### 阶段 3：快速实现（30 分钟）

**你 → AI**：
```
明白了，现在请帮我实现：

1. 先创建 index.html，包含基本结构
2. 要求：
   - 简洁的界面
   - 输入框和按钮
   - 任务列表区域
```

**AI 生成代码** → **你立即查看**：
```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>任务管理器</title>
    <link rel="stylesheet" href="style.css">
</head>
<body>
    <div class="container">
        <h1>我的任务</h1>
        <div class="input-section">
            <input type="text" id="taskInput" placeholder="添加新任务...">
            <button id="addBtn">添加</button>
        </div>
        <ul id="taskList"></ul>
    </div>
    <script src="app.js"></script>
</body>
</html>
```

**你的判断**：
- ✅ 结构看起来对
- ✅ ID 命名清晰
- ⚠️ 但我想要更好看的界面

**你 → AI**：
```
HTML 结构不错，现在请创建 style.css，要求：
- 现代化的设计
- 居中布局
- 任务项有悬停效果
- 完成的任务有删除线
```

**AI 生成 CSS** → **你保存并在浏览器打开**

**你看到的效果**：
- ✅ 界面还不错
- ✅ 布局居中
- 🎯 **第一次反馈**：能看到页面了！

**你 → AI**：
```
界面不错！现在请实现 app.js，功能：
1. 添加任务
2. 显示任务列表
3. 点击任务切换完成状态
4. 删除按钮

先实现基础功能，不用考虑 localStorage。
```

**AI 生成 app.js** → **你保存并刷新页面**

**你测试**：
1. 输入"买菜" → 点击添加 → ✅ 出现在列表中
2. 点击任务 → ✅ 出现删除线
3. 点击删除 → ✅ 任务消失

**🎯 第二次反馈**：基础功能能用了！

#### 阶段 4：迭代优化（20 分钟）

**你发现的问题**：
1. 刷新页面后任务消失了
2. 按回车键不能添加任务
3. 删除按钮不够明显

**你 → AI**：
```
基础功能能用了，但有几个问题：

1. 添加 localStorage 持久化
2. 支持按回车键添加任务
3. 删除按钮改成红色，更明显

请修改 app.js 和 style.css。
```

**AI 修改代码** → **你测试**：
1. 添加任务 → 刷新页面 → ✅ 任务还在
2. 输入任务 → 按回车 → ✅ 添加成功
3. 删除按钮 → ✅ 红色，更明显

**🎯 第三次反馈**：完美！

#### 阶段 5：最后润色（10 分钟）

**你 → AI**：
```
功能都好了，最后优化一下：

1. 添加任务后自动清空输入框
2. 添加任务后自动聚焦输入框
3. 空任务不能添加
4. 添加一个"清空所有"按钮
```

**AI 修改代码** → **你测试** → ✅ 全部通过

**🎯 最终结果**：
- ⏱️ 总用时：约 75 分钟
- ✅ 功能完整
- ✅ 界面美观
- ✅ 体验良好

### 2.3 逐步解析每个环节

#### 环节 1：理解需求

**关键点**：
- ❌ 不要直接要求 AI 写代码
- ✅ 先让 AI 帮你理解需求
- ✅ 明确核心功能和可选功能
- ✅ 选择合适的技术栈

**为什么重要**：
- 避免做了才发现方向错了
- 理解了需求才能判断 AI 的输出
- 选对技术栈事半功倍

#### 环节 2：技术方案

**关键点**：
- ❌ 不要让 AI 直接写代码
- ✅ 先理解整体结构
- ✅ 明确文件组织
- ✅ 了解关键技术点

**为什么重要**：
- 有了整体理解，才能判断细节
- 知道"为什么这样做"
- 遇到问题知道从哪里查

#### 环节 3：快速实现

**关键点**：
- ✅ 分步实现（HTML → CSS → JS）
- ✅ 每一步都立即验证
- ✅ 先实现核心功能
- ✅ 不追求完美

**为什么重要**：
- 快速看到结果，保持动力
- 及早发现问题
- 避免一次做太多

#### 环节 4：迭代优化

**关键点**：
- ✅ 基于实际使用发现问题
- ✅ 一次解决 2-3 个问题

**为什么重要**：
- 真实使用才能发现真实问题

- 持续改进，逐步完美

#### 环节 5：最后润色

**关键点**：

**为什么重要**：
- 细节决定体验

## 3. 建立基本工作流

通过第一个项目，你已经体验了 Vibe Coding。现在让我们总结出可复用的工作流。

### 3.1 描述需求 → AI 建议 → 判断 → 迭代

这是 Vibe Coding 的核心循环，理解并掌握它至关重要。

#### 步骤 1：描述需求

**好的需求描述**：

✅ **清晰但不过度具体**
```
好："我想做一个任务管理器，能添加、完成、删除任务"
差："帮我做一个应用"（太模糊）
差："用 React + Redux + TypeScript + Tailwind 实现一个任务管理器，
     要有 useReducer hook，状态要用 Immer 管理..."（过度具体）
```

✅ **说明目标而非实现**
```
好："我想让用户能快速添加任务"
差："给输入框加一个回车事件监听器"
```

✅ **提供上下文**
```
好："我在做一个个人项目，想快速验证想法，代码质量要求不高"
差："帮我写代码"
```

✅ **说明约束条件**
```
好："我不熟悉 React，希望用纯 JavaScript"
好："这个功能会被频繁调用，需要考虑性能"
```

**需求描述模板**：
```
我想做 [功能描述]

背景：[为什么要做这个]
目标：[期望达到什么效果]
约束：[技术栈、时间、质量要求等]
不确定的地方：[哪些地方需要建议]
```

#### 步骤 2：AI 建议

**AI 可能给出的建议类型**：

**类型 1：方案建议**
```
"有三种实现方案：
A. 用 localStorage（简单，但数据量有限）
B. 用 IndexedDB（复杂，但功能强大）
C. 用后端 API（需要服务器）

推荐：如果是个人项目，用 A"
```

**类型 2：技术建议**
```
"这个场景适合用 React Hooks 的 useState 和 useEffect"
```

**类型 3：实现建议**
```
"建议先实现核心功能，再考虑优化"
```

**如何获得更好的建议**：

✅ **明确要求建议**
```
"请给我 2-3 个方案，并说明优缺点"
```

✅ **要求解释理由**
```
"为什么推荐这个方案？"
```

✅ **要求考虑你的情况**
```
"考虑到我是初学者，哪个方案更合适？"
```

#### 步骤 3：判断

这是最关键的一步，决定了 Vibe Coding 的质量。

**判断的维度**：

**1. 方向判断**
```
问自己：
- 这个方案符合我的目标吗？
- 是否偏离了初衷？
- 是否过度复杂了？
```

**2. 可行性判断**
```
问自己：
- 我能理解这个方案吗？
- 我能实现吗？
- 有没有明显的问题？
```

**3. 优先级判断**
```
问自己：
- 这个现在必须做吗？
- 能不能先做更重要的？
- 这个可以以后再优化吗？
```

**判断的方法**：

**方法 1：快速原型验证**
```
不确定方案是否可行？
→ 让 AI 生成一个最小示例
→ 运行看看效果
→ 再决定是否采用
```

**方法 2：对比多个方案**
```
不确定哪个方案更好？
→ 让 AI 给出 2-3 个方案
→ 对比优缺点
→ 选择最适合的
```

**方法 3：咨询 AI**
```
不确定如何判断？
→ 问 AI："这个方案有什么潜在问题？"
→ 问 AI："有没有更简单的方案？"
→ 基于 AI 的回答做判断
```

**判断的原则**：

✅ **相信直觉**
- 感觉不对，就是不对
- 不要说服自己接受不舒服的方案

✅ **但要验证直觉**
- 直觉可能错
- 用实际运行来验证

✅ **保持怀疑**
- AI 可能错
- 关键决策要自己判断

✅ **但不要过度怀疑**
- 不是每个细节都要质疑
- 信任但验证

#### 步骤 4：迭代

**迭代的类型**：

**类型 1：修正迭代**
```
发现问题 → 修复 → 验证

例子：
"这个按钮点击没反应"
→ 让 AI 修复
→ 测试，确认修复
```

**类型 2：优化迭代**
```
基础功能 OK → 优化体验 → 验证

例子：
"功能能用了，但界面不够美观"
→ 让 AI 优化样式
→ 查看效果
```

**类型 3：扩展迭代**
```
核心功能 OK → 添加新功能 → 验证

例子：
"基础任务管理能用了，现在想添加分类功能"
→ 让 AI 实现
→ 测试新功能
```

**迭代的节奏**：

**快速迭代**（推荐）：
```
小改动 → 立即验证 → 再改动
周期：5-10 分钟
```

**批量迭代**（不推荐）：
```
多个改动 → 一起验证
周期：30-60 分钟
问题：出错了不知道哪里错
```

**迭代的停止条件**：

✅ **达到目标**
- 功能实现了
- 效果满意了

✅ **投入产出比不划算**
- 继续优化收益很小
- 时间可以用在更重要的地方

✅ **遇到瓶颈**
- 当前方案走不通
- 需要换方向

❌ **不好的停止理由**：
- "感觉差不多了"（太模糊）
- "累了"（不是理性判断）
- "时间到了"（忽视了目标）

### 3.2 如何判断"感觉对不对"

"感觉"不是玄学，而是快速的整体判断。如何培养这种感觉？

#### 维度 1：代码结构

**好的感觉**：
- ✅ 代码结构清晰
- ✅ 命名容易理解
- ✅ 逻辑流程顺畅
- ✅ 没有明显的重复

**不好的感觉**：
- ❌ 代码很乱，难以理解
- ❌ 命名莫名其妙
- ❌ 逻辑跳来跳去
- ❌ 大量重复代码

**如何判断**：
```
问自己：
- 我能快速理解这段代码在做什么吗？
- 如果一个月后回来看，我还能理解吗？
- 如果要修改，我知道从哪里改吗？
```

#### 维度 2：功能实现

**好的感觉**：
- ✅ 功能符合预期
- ✅ 没有明显的 bug
- ✅ 边界情况处理了
- ✅ 用户体验流畅

**不好的感觉**：
- ❌ 功能不对
- ❌ 经常出错
- ❌ 边界情况崩溃
- ❌ 用起来别扭

**如何判断**：
```
实际使用：
- 像真实用户一样使用
- 尝试各种操作
- 看看有没有问题
```

#### 维度 3：技术方案

**好的感觉**：
- ✅ 方案简单直接
- ✅ 没有过度设计
- ✅ 容易扩展
- ✅ 性能够用

**不好的感觉**：
- ❌ 方案过于复杂
- ❌ 为了"优雅"牺牲简单
- ❌ 难以修改
- ❌ 性能明显有问题

**如何判断**：
```
问自己：
- 这个方案是最简单的吗？
- 有没有过度设计？
- 如果需求变化，容易改吗？
```

#### 维度 4：整体协调

**好的感觉**：
- ✅ 各部分配合良好
- ✅ 风格统一
- ✅ 没有明显的不协调

**不好的感觉**：
- ❌ 各部分风格不一致
- ❌ 有些地方很精致，有些很粗糙
- ❌ 整体不和谐

**如何判断**：
```
整体审视：
- 退后一步，看整体
- 有没有明显的不协调？
- 哪里看起来不对？
```

#### 培养"感觉"的方法

**方法 1：多实践**
- 做得越多，感觉越准
- 每次都验证自己的感觉
- 总结什么样的"感觉"是对的

**方法 2：对比学习**
- 看优秀的代码
- 对比自己的代码
- 理解差距在哪里

**方法 3：及时反馈**
- 每次判断后立即验证
- 对了，强化这种感觉
- 错了，调整判断标准

**方法 4：建立检查清单**
```
我的检查清单：
□ 代码能看懂吗？
□ 功能对吗？
□ 有明显的问题吗？
□ 整体协调吗？
□ 我满意吗？
```

## 4. 新手常犯的 5 个错误

从实践中总结的最常见错误，避免这些能让你少走很多弯路。

### 错误 1：让 AI 全盘接管

#### 错误示例

```
用户："帮我做一个电商网站，要有用户系统、商品管理、
      订单系统、支付功能、后台管理..."

AI：生成 10000 行代码

用户：直接复制粘贴，运行
      → 报错一堆
      → 不知道从哪里改
      → 放弃
```

#### 为什么错

- ❌ 需求太大，AI 无法一次做好
- ❌ 你不理解代码，无法判断和修改
- ❌ 出问题不知道怎么办

#### 正确做法

```
用户："我想做电商网站，先帮我理解需要哪些模块"

AI：解释模块划分

用户："明白了，我们先做商品展示，其他以后再说"

AI：生成商品展示代码

用户：运行，测试，理解

用户："好了，现在加上购物车"

AI：添加购物车功能

...（逐步迭代）
```

**关键原则**：
- ✅ 小步快跑
- ✅ 每一步都理解
- ✅ 逐步构建

### 错误 2：从不验证就继续

#### 错误示例

```
用户："帮我写登录功能"
AI：生成代码
用户："好，现在写注册功能"
AI：生成代码
用户："好，现在写找回密码"
AI：生成代码
用户："好，现在写个人中心"
AI：生成代码

...（写了一堆）

用户：第一次运行
      → 登录功能就不对
      → 后面全白做了
```

#### 为什么错

- ❌ 基础错了，后面全错
- ❌ 问题积累，难以定位
- ❌ 浪费时间

#### 正确做法

```
用户："帮我写登录功能"
AI：生成代码
用户：立即运行测试
      → 发现问题
      → 让 AI 修复
      → 再测试
      → 确认 OK

用户："好了，现在写注册功能"
AI：生成代码
用户：立即测试
      → 确认 OK

...（每一步都验证）
```

**关键原则**：
- ✅ 写一点，测一点
- ✅ 确认 OK 再继续
- ✅ 及早发现问题

### 错误 3：盲目接受 AI 建议

#### 错误示例

```
用户："我想做一个简单的计数器"

AI："建议使用 Redux + Redux-Saga + TypeScript + 
     Immutable.js，这样架构更清晰，可维护性更好..."

用户："好的"（虽然完全不懂）
      → 花了 3 天学习这些技术
      → 最后发现一个 useState 就够了
```

#### 为什么错

- ❌ AI 可能过度设计
- ❌ 不适合你的场景
- ❌ 浪费时间学习不必要的东西

#### 正确做法

```
用户："我想做一个简单的计数器"

AI："建议使用 Redux..."

用户："等等，为什么需要 Redux？我只是想做一个简单的计数器"

AI："对于简单计数器，useState 就够了。
     Redux 适合复杂的状态管理"

用户："那就用 useState"
```

**关键原则**：
- ✅ 质疑复杂的方案
- ✅ 问"为什么需要这个"
- ✅ 选择最简单的方案

### 错误 4：追求完美的第一版

#### 错误示例

```
用户："帮我做任务管理器"

AI：生成基础版本

用户："等等，这个界面不够好看，重新设计"
AI：重新设计

用户："这个动画不够流畅，加上过渡效果"
AI：添加动画

用户："这个颜色不对，换一个配色方案"
AI：换配色

用户："这个字体不好，换一个"
AI：换字体

...（一直在调整细节）

结果：花了 2 天，还没有完整功能
```

#### 为什么错

- ❌ 陷入细节，忘记目标
- ❌ 完美主义，永远不满意
- ❌ 浪费时间在不重要的地方

#### 正确做法

```
用户："帮我做任务管理器"

AI：生成基础版本

用户：测试功能
      → 能添加任务 ✅
      → 能完成任务 ✅
      → 能删除任务 ✅
      → 功能都对了

用户："好，第一版完成。现在优化界面"

AI：优化界面

用户："够用了，先这样。以后有需要再优化"
```

**关键原则**：
- ✅ 先实现功能
- ✅ 再优化体验
- ✅ 知道何时"够了"

### 错误 5：不记录决策过程

#### 错误示例

```
第 1 天：
用户："为什么用 localStorage 而不是 IndexedDB？"
AI：解释原因
用户："明白了"

第 7 天：
用户："为什么当初用 localStorage？现在数据太多了"
用户：不记得当初的考虑
      → 不知道是否应该换
      → 不知道换的影响
```

#### 为什么错

- ❌ 忘记当初的决策理由
- ❌ 重复犯同样的错误
- ❌ 难以维护和扩展

#### 正确做法

```
第 1 天：
用户："为什么用 localStorage 而不是 IndexedDB？"
AI：解释原因
用户：在代码注释或文档中记录：
      "使用 localStorage 因为：
       1. 数据量小（< 5MB）
       2. 不需要复杂查询
       3. 简单易用
       如果数据量增大，考虑迁移到 IndexedDB"

第 7 天：
用户：查看记录
      → 明白当初的考虑
      → 知道现在该换了
      → 知道如何迁移
```

**关键原则**：
- ✅ 记录重要决策
- ✅ 记录决策理由
- ✅ 记录未来可能的调整

**记录方式**：
```
方式 1：代码注释
// 使用 localStorage 而不是 IndexedDB
// 原因：数据量小，不需要复杂查询
// 限制：最多 5MB 数据
// 如果超过限制，迁移到 IndexedDB

方式 2：README 文档
## 技术决策

### 数据存储
- 方案：localStorage
- 理由：...
- 限制：...
- 未来：...

方式 3：Git 提交信息
git commit -m "使用 localStorage 存储任务

选择 localStorage 而不是 IndexedDB：
- 数据量小
- 不需要复杂查询
- 简单易用"
```
