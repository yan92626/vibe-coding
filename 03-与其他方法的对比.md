## 1. Vibe Coding vs 传统瀑布开发

### 1.1 瀑布开发简介

**瀑布模型**是最传统的软件开发方法，像瀑布一样单向流动：

```
需求分析 → 系统设计 → 实现 → 测试 → 部署 → 维护
```

**核心特点**：
- 线性流程，不可逆
- 每个阶段完成后才进入下一阶段
- 强调文档和规划
- 变更成本高

### 1.2 核心差异

| 维度 | 瀑布开发 | Vibe Coding |
|------|----------|-------------|
| **流程** | 线性、单向 | 循环、迭代 |
| **起点** | 完整的需求文档 | 模糊的想法 |
| **规划** | 详细的前期规划 | 边做边规划 |
| **变更** | 变更需要走流程，成本高 | 变更是常态，成本低 |
| **反馈** | 测试阶段才有反馈 | 持续反馈 |
| **文档** | 大量文档 | 最小化文档 |
| **风险** | 后期发现问题，风险集中 | 早期发现问题，风险分散 |

### 1.3 优劣对比

#### 瀑布开发的优势

✅ **适合需求明确的项目**
- 需求稳定，不会变化
- 例如：政府项目、合同项目

✅ **可预测性强**
- 时间、成本、范围都明确
- 便于管理和控制

✅ **文档完整**
- 便于交接和维护
- 符合合规要求

#### 瀑布开发的劣势

❌ **不适应变化**
- 需求变了，整个流程要重来
- 变更成本极高

❌ **反馈太晚**
- 做完才知道对不对
- 错了代价巨大

❌ **速度慢**
- 前期规划时间长
- 交付周期长

#### Vibe Coding 的优势

✅ **快速适应变化**
- 需求变了，立即调整
- 变更成本低

✅ **快速反馈**
- 边做边验证
- 及早发现问题

✅ **速度快**
- 快速开始
- 快速交付

#### Vibe Coding 的劣势

❌ **不适合高风险项目**
- 关键系统需要严格规划
- 不能"边做边看"

❌ **可预测性差**
- 难以准确估算时间
- 范围可能变化

❌ **文档不足**
- 可能缺少完整文档
- 不符合某些合规要求

### 1.4 何时选择哪种方法

**选择瀑布开发**：
- ✅ 需求非常明确且稳定
- ✅ 有严格的合规要求
- ✅ 项目风险极高
- ✅ 需要详细的文档
- ✅ 团队习惯传统流程

**选择 Vibe Coding**：
- ✅ 需求模糊或可能变化
- ✅ 需要快速验证想法
- ✅ 项目风险可控
- ✅ 速度比完美更重要
- ✅ 团队接受新方法

**混合使用**：
```
整体项目管理 → 瀑布（明确里程碑）
具体功能开发 → Vibe Coding（快速迭代）
```

### 1.5 实际案例对比

#### 案例：开发一个电商网站

**瀑布方法**：
```
第 1-2 周：需求分析（写 50 页需求文档）
第 3-4 周：系统设计（画架构图、数据库设计）
第 5-10 周：编码实现
第 11-12 周：测试
第 13 周：部署

结果：13 周后上线，发现用户不喜欢某些功能
```

**Vibe Coding 方法**：
```
第 1 天：做出商品展示页面原型
第 2 天：加上购物车功能
第 3 天：实现简单支付
第 4 天：给用户试用，收集反馈
第 5-7 天：根据反馈调整
第 2 周：上线 MVP

结果：2 周上线，根据真实反馈持续优化
```

## 2. Vibe Coding vs 敏捷开发

### 2.1 敏捷开发简介

**敏捷开发**是对瀑布模型的改进，强调迭代和适应变化：

**核心价值观**（敏捷宣言）：
- 个体和互动 > 流程和工具
- 工作的软件 > 详尽的文档
- 客户合作 > 合同谈判
- 响应变化 > 遵循计划

**常见实践**：
- Scrum（冲刺、站会、回顾）
- 用户故事
- 持续集成
- 小步迭代

### 2.2 核心差异

| 维度 | 敏捷开发 | Vibe Coding |
|------|----------|-------------|
| **迭代周期** | 1-4 周的 Sprint | 分钟到小时级的迭代 |
| **团队** | 强调团队协作 | 可以个人使用 |
| **流程** | 有明确的流程和仪式 | 流程灵活，最小化 |
| **规划** | Sprint 规划会议 | 边做边规划 |
| **AI 角色** | 工具辅助 | 核心协作伙伴 |
| **文档** | 最小化但仍需要 | 更少或无文档 |
| **适用场景** | 团队项目 | 个人或小团队 |

### 2.3 相似之处

Vibe Coding 和敏捷开发有很多共同点：

✅ **都强调迭代**
- 小步快跑
- 持续改进

✅ **都拥抱变化**
- 需求可以变
- 根据反馈调整

✅ **都重视反馈**
- 快速验证
- 及早发现问题

✅ **都追求"工作的软件"**
- 能跑的代码 > 完美的文档
- 实际价值 > 形式主义

### 2.4 关键区别

#### 区别 1：迭代粒度

**敏捷**：
- Sprint（1-4 周）
- 每个 Sprint 交付可用功能
- Sprint 内不改变目标

**Vibe Coding**：
- 微迭代（分钟到小时）
- 每次迭代验证一个小假设
- 随时可以调整方向

#### 区别 2：团队 vs 个人

**敏捷**：
- 为团队协作设计
- 需要站会、回顾等仪式
- 强调沟通和协作

**Vibe Coding**：
- 可以个人使用
- 不需要团队仪式
- 人机协作为主

#### 区别 3：AI 的角色

**敏捷**：
- AI 是辅助工具
- 人仍然写大部分代码
- 流程不依赖 AI

**Vibe Coding**：
- AI 是核心协作伙伴
- AI 生成大部分代码
- 方法依赖 AI 能力

### 2.5 两者的结合

Vibe Coding 可以作为敏捷开发的**实现方式**：

```
敏捷框架（宏观）
  ├─ Sprint 规划（团队层面）
  ├─ 用户故事（需求层面）
  └─ 具体实现 → Vibe Coding（个人层面）
      ├─ 快速迭代
      ├─ AI 协作
      └─ 持续反馈
```

**实践建议**：
- 用敏捷管理团队和项目
- 用 Vibe Coding 实现具体功能
- 两者互补，不冲突

### 2.6 实际案例对比

#### 案例：开发一个任务管理功能

**敏捷方法**：
```
Sprint 规划：
- 用户故事：作为用户，我想创建任务
- 估算：3 个故事点
- 分配：开发者 A

开发过程：
- 第 1-2 天：设计和编码
- 第 3 天：代码审查
- 第 4 天：测试
- 第 5 天：集成

Sprint 结束：演示功能
```

**Vibe Coding 方法**：
```
第 1 小时：
- 和 AI 讨论任务管理的实现方案
- 生成基础的任务列表组件
- 运行，看效果

第 2 小时：
- 添加创建任务功能
- 测试，发现 UI 不好用
- 调整布局

第 3 小时：
- 添加编辑和删除
- 整体测试
- 完成

结果：3 小时完成，期间持续验证
```

**结合使用**：
```
Sprint 层面：用敏捷
- Sprint 目标：完成任务管理模块
- 用户故事：创建、编辑、删除任务

实现层面：用 Vibe Coding
- 每个故事用 Vibe Coding 快速实现
- 持续迭代和优化
- 快速响应反馈

结果：敏捷的结构 + Vibe Coding 的速度
```

## 3. Vibe Coding vs TDD（测试驱动开发）

### 3.1 TDD 简介

**测试驱动开发（TDD）**是一种开发方法，核心是"先写测试，再写代码"：

**TDD 循环**（红-绿-重构）：
```
1. 写一个失败的测试（红）
2. 写最少的代码让测试通过（绿）
3. 重构代码（重构）
4. 重复
```

**核心理念**：
- 测试定义需求
- 测试保证质量
- 测试驱动设计

### 3.2 核心差异

| 维度        | TDD     | Vibe Coding |
| --------- | ------- | ----------- |
| **起点**    | 测试用例    | 需求描述        |
| **驱动力**   | 测试      | 反馈          |
| **验证方式**  | 自动化测试   | 运行和观察       |
| **重点**    | 正确性     | 快速验证        |
| **适用场景**  | 逻辑复杂的代码 | 快速原型和探索     |
| **学习曲线**  | 陡峭      | 平缓          |
| **AI 角色** | 辅助写测试   | 核心协作伙伴      |

### 3.3 优劣对比

#### TDD 的优势

✅ **高质量代码**
- 测试覆盖率高
- Bug 少
- 可维护性好

✅ **清晰的需求**
- 测试即文档
- 需求明确

✅ **重构信心**
- 有测试保护
- 敢于重构

#### TDD 的劣势

❌ **速度慢**
- 写测试需要时间
- 前期投入大

❌ **学习曲线陡**
- 需要改变思维方式
- 需要测试技能

❌ **不适合探索**
- 需求不明确时难以写测试
- 频繁变化时测试也要改

#### Vibe Coding 的优势

✅ **速度快**
- 不需要先写测试
- 快速看到结果

✅ **适合探索**
- 需求模糊时也能开始
- 快速试错

✅ **学习曲线平缓**
- 不需要测试技能
- 自然的工作方式

#### Vibe Coding 的劣势

❌ **质量保证弱**
- 没有自动化测试
- 可能遗漏边界情况

❌ **重构风险高**
- 没有测试保护
- 容易引入 bug

❌ **不适合复杂逻辑**
- 复杂逻辑需要严格验证
- 手动测试不够

### 3.4 两者的结合

TDD 和 Vibe Coding 可以互补：

**策略 1：先 Vibe，后 TDD**
```
1. 用 Vibe Coding 快速探索和原型
2. 确定方案后，用 TDD 重写
3. 得到高质量的生产代码
```

**策略 2：分层使用**
```
核心逻辑 → TDD（保证正确性）
UI 和交互 → Vibe Coding（快速迭代）
工具脚本 → Vibe Coding（一次性任务）
```

**策略 3：AI 辅助 TDD**
```
1. 描述需求给 AI
2. 让 AI 生成测试用例
3. 让 AI 生成实现代码
4. 运行测试，迭代优化

结果：TDD 的质量 + Vibe Coding 的速度
```

### 3.5 实际案例对比

#### 案例：实现一个计算器函数

**TDD 方法**：
```
1. 写测试：
   test('add(1, 2) should return 3')
   test('add(-1, 1) should return 0')
   test('add(0, 0) should return 0')

2. 写实现：
   function add(a, b) { return a + b }

3. 运行测试：全部通过

4. 重复其他运算符

时间：1-2 小时（包括写测试）
质量：高（有完整测试）
```

**Vibe Coding 方法**：
```
1. 告诉 AI："写一个计算器函数，支持加减乘除"

2. AI 生成代码

3. 手动测试几个例子

4. 发现问题，让 AI 修复

5. 再测试，通过

时间：10-15 分钟
质量：中（没有完整测试）
```

**结合方法**：
```
1. 用 Vibe Coding 快速生成实现

2. 让 AI 生成测试用例

3. 运行测试，发现问题

4. 让 AI 修复

5. 测试通过

时间：20-30 分钟
质量：高（有 AI 生成的测试）
```

## 4. Vibe Coding vs 结对编程

### 4.1 结对编程简介

**结对编程（Pair Programming）**是两个程序员一起工作：

**角色**：
- **驾驶员（Driver）**：写代码
- **导航员（Navigator）**：审查、思考、指导

**核心理念**：
- 两个人比一个人更好
- 实时代码审查
- 知识共享

### 4.2 核心差异

| 维度 | 结对编程 | Vibe Coding |
|------|----------|-------------|
| **协作对象** | 另一个人 | AI |
| **角色** | 驾驶员 + 导航员 | 人（决策）+ AI（执行） |
| **沟通** | 人与人对话 | 人与 AI 对话 |
| **成本** | 2 个人的时间 | 1 个人 + AI |
| **知识共享** | 双向学习 | 从 AI 学习 |
| **可用性** | 需要另一个人 | 随时可用 |

### 4.3 相似之处

✅ **都是协作模式**
- 不是一个人独自工作
- 有对话和交流

✅ **都有实时反馈**
- 立即发现问题
- 立即讨论方案

✅ **都促进思考**
- 需要解释自己的想法
- 需要理解对方的建议

✅ **都提升质量**
- 多一双眼睛审查
- 减少错误

### 4.4 关键区别

#### 区别 1：协作对象的能力

**结对编程（人）**：
- ✅ 理解业务逻辑
- ✅ 创造性思维
- ✅ 判断力和经验
- ❌ 知识有限
- ❌ 可能疲劳

**Vibe Coding（AI）**：
- ✅ 知识广博
- ✅ 不知疲倦
- ✅ 快速执行
- ❌ 不理解业务
- ❌ 可能出错

#### 区别 2：成本和可用性

**结对编程**：
- 需要 2 个人的时间
- 需要协调时间
- 不是随时可用

**Vibe Coding**：
- 只需要 1 个人的时间
- 随时可用
- 不需要协调

#### 区别 3：学习效果

**结对编程**：
- 双向学习
- 学习对方的思维方式
- 学习领域知识

**Vibe Coding**：
- 单向学习（从 AI）
- 学习技术知识
- 学习实现模式

### 4.5 两者的结合

**策略 1：人人结对 + AI 辅助**
```
两个人结对编程
  ├─ 讨论方案（人与人）
  ├─ 让 AI 生成代码（人与 AI）
  └─ 共同审查（人与人）

结果：人的判断 + AI 的速度
```

**策略 2：不同场景不同方法**
```
复杂业务逻辑 → 结对编程（需要人的理解）
标准功能实现 → Vibe Coding（AI 足够）
学习新技术 → Vibe Coding（AI 是好老师）
关键决策 → 结对编程（需要讨论）
```

### 4.6 实际案例对比

#### 案例：实现一个复杂的业务规则

**结对编程**：
```
导航员："这个规则是，如果用户是 VIP 且订单超过 1000 元..."
驾驶员："明白，我写一个 if 判断"
导航员："等等，还要考虑优惠券的情况"
驾驶员："对，我加上"
导航员："这里逻辑有点复杂，要不要提取一个函数？"
驾驶员："好主意"

结果：逻辑正确，结构清晰
时间：30 分钟
```

**Vibe Coding**：
```
用户："实现 VIP 折扣规则，订单超过 1000 元打 9 折"
AI：生成代码
用户："还要考虑优惠券"
AI：修改代码
用户："逻辑有点乱，重构一下"
AI：重构代码

结果：功能实现，但可能遗漏边界情况
时间：15 分钟
```

**结合方法**：
```
两个人 + AI：
人 A："我们需要实现 VIP 折扣规则"
人 B："要考虑哪些情况？"
人 A："VIP、订单金额、优惠券..."
人 B："让 AI 先生成一个版本"
AI：生成代码
人 A："这里的逻辑对吗？"
人 B："有个边界情况没考虑"
人 A："让 AI 修复"
AI：修复代码
人 B："现在看起来对了"

结果：逻辑正确 + 快速实现
时间：20 分钟
```

## 5. 综合对比表

### 5.1 全面对比矩阵

| 维度 | 瀑布开发 | 敏捷开发 | TDD | 结对编程 | Vibe Coding |
|------|----------|----------|-----|----------|-------------|
| **迭代周期** | 月/年 | 周 | 小时/天 | 实时 | 分钟/小时 |
| **规划程度** | 详细前期规划 | Sprint 规划 | 测试即规划 | 边做边规划 | 最小规划 |
| **文档要求** | 大量文档 | 最小化文档 | 测试即文档 | 口头交流 | 极少文档 |
| **变更成本** | 极高 | 中等 | 低 | 低 | 极低 |
| **反馈速度** | 慢（月） | 中（周） | 快（小时） | 实时 | 极快（分钟） |
| **团队要求** | 多角色团队 | 跨职能团队 | 开发者 | 2 个开发者 | 1 个人 + AI |
| **学习曲线** | 平缓 | 中等 | 陡峭 | 平缓 | 平缓 |
| **质量保证** | 测试阶段 | 持续测试 | 测试驱动 | 实时审查 | 人工验证 |
| **适用规模** | 大型项目 | 中大型项目 | 任何规模 | 任何规模 | 小中型项目 |
| **风险管理** | 后期集中 | 分散到 Sprint | 持续降低 | 实时发现 | 快速试错 |
| **创新能力** | 低 | 中 | 中 | 高 | 极高 |
| **可预测性** | 高 | 中 | 中 | 低 | 低 |

### 5.2 场景适用性对比

| 场景 | 瀑布 | 敏捷 | TDD | 结对 | Vibe |
|------|------|------|-----|------|------|
| **快速原型** | ❌ | ⚠️ | ❌ | ⚠️ | ✅ |
| **学习新技术** | ❌ | ⚠️ | ❌ | ✅ | ✅ |
| **关键业务系统** | ✅ | ✅ | ✅ | ✅ | ❌ |
| **个人项目** | ❌ | ⚠️ | ⚠️ | ❌ | ✅ |
| **团队协作** | ✅ | ✅ | ✅ | ✅ | ⚠️ |
| **高性能代码** | ✅ | ⚠️ | ✅ | ✅ | ❌ |
| **探索性编程** | ❌ | ⚠️ | ❌ | ⚠️ | ✅ |
| **合规项目** | ✅ | ⚠️ | ✅ | ⚠️ | ❌ |
| **快速迭代** | ❌ | ✅ | ⚠️ | ⚠️ | ✅ |
| **复杂逻辑** | ✅ | ✅ | ✅ | ✅ | ⚠️ |

**图例**：
- ✅ 非常适合
- ⚠️ 可以使用，但有限制
- ❌ 不适合

### 5.3 成本效益对比

#### 时间成本

**从想法到可用原型**：
```
瀑布：2-3 个月
敏捷：2-4 周
TDD：1-2 周
结对：1 周
Vibe Coding：1-3 天
```

**从原型到生产**：
```
瀑布：6-12 个月
敏捷：2-6 个月
TDD：1-3 个月
结对：1-2 个月
Vibe Coding：2-4 周（需要质量提升）
```

#### 人力成本

**小型项目（个人可完成）**：
```
瀑布：1 人 × 3 个月 = 3 人月
敏捷：1 人 × 2 个月 = 2 人月
TDD：1 人 × 1.5 个月 = 1.5 人月
结对：2 人 × 1 个月 = 2 人月
Vibe Coding：1 人 × 2 周 = 0.5 人月
```

**中型项目（团队协作）**：
```
瀑布：5 人 × 6 个月 = 30 人月
敏捷：5 人 × 4 个月 = 20 人月
TDD：5 人 × 3 个月 = 15 人月
结对：10 人 × 2 个月 = 20 人月
Vibe Coding：3 人 × 2 个月 = 6 人月
```

#### 质量成本

**Bug 修复成本**（相对值）：
```
瀑布：100（后期发现，修复成本高）
敏捷：50（Sprint 内发现）
TDD：10（测试驱动，bug 少）
结对：20（实时审查）
Vibe Coding：30（快速发现，但可能遗漏）
```

**维护成本**（相对值）：
```
瀑布：50（文档完整，但代码可能过时）
敏捷：40（持续更新）
TDD：20（测试保护，易维护）
结对：30（代码质量高）
Vibe Coding：60（文档少，依赖原作者）
```

### 5.4 优势劣势总结

#### 瀑布开发

**最大优势**：可预测性、文档完整、适合合规
**最大劣势**：不适应变化、反馈慢、速度慢
**最适合**：需求明确、风险极高、合规要求严格的项目

#### 敏捷开发

**最大优势**：平衡速度和质量、适应变化、团队协作
**最大劣势**：需要团队配合、有一定流程开销
**最适合**：需求会变化、团队协作的中大型项目

#### TDD

**最大优势**：代码质量高、重构信心、测试覆盖
**最大劣势**：学习曲线陡、前期投入大、不适合探索
**最适合**：逻辑复杂、需要长期维护的核心代码

#### 结对编程

**最大优势**：实时审查、知识共享、质量高
**最大劣势**：成本高（2 人）、需要协调
**最适合**：复杂业务逻辑、知识传递、关键功能

#### Vibe Coding

**最大优势**：速度极快、适合探索、学习曲线平缓
**最大劣势**：质量保证弱、不适合高风险场景
**最适合**：快速原型、个人项目、学习新技术

## 6. 方法选择指南

### 6.1 决策树

```
开始
├─ 需求是否明确且稳定？
│  ├─ 是 → 是否有严格合规要求？
│  │  ├─ 是 → 瀑布开发
│  │  └─ 否 → 是否团队协作？
│  │     ├─ 是 → 敏捷开发
│  │     └─ 否 → TDD 或 Vibe Coding
│  └─ 否 → 是否需要快速验证？
│     ├─ 是 → Vibe Coding
│     └─ 否 → 敏捷开发
```

### 6.2 按项目特征选择

#### 项目规模

**小型项目**（个人或 2-3 人）：
- 首选：Vibe Coding
- 备选：敏捷（如果需要流程）

**中型项目**（5-10 人）：
- 首选：敏捷 + Vibe Coding
- 备选：敏捷 + TDD

**大型项目**（10+ 人）：
- 首选：敏捷
- 备选：瀑布（如果需求极其明确）

#### 项目风险

**低风险**（失败影响小）：
- 首选：Vibe Coding
- 备选：敏捷

**中风险**（失败有影响但可控）：
- 首选：敏捷
- 备选：敏捷 + TDD

**高风险**（失败影响严重）：
- 首选：瀑布 或 敏捷 + TDD
- 备选：结对编程

#### 项目类型

**原型/MVP**：
- 首选：Vibe Coding
- 目标：快速验证

**产品开发**：
- 首选：敏捷
- 目标：平衡速度和质量

**关键系统**：
- 首选：瀑布 或 敏捷 + TDD
- 目标：质量和可靠性

**个人项目**：
- 首选：Vibe Coding
- 目标：快速实现想法

### 6.3 混合使用策略

#### 策略 1：分层混合

```
项目管理层 → 敏捷（Sprint、站会）
架构设计层 → 瀑布（详细规划）
功能实现层 → Vibe Coding（快速迭代）
核心逻辑层 → TDD（质量保证）
复杂功能层 → 结对编程（知识共享）
```

#### 策略 2：阶段混合

```
探索阶段 → Vibe Coding（快速试错）
原型阶段 → Vibe Coding（快速验证）
开发阶段 → 敏捷 + Vibe Coding（平衡）
优化阶段 → TDD（质量提升）
维护阶段 → 敏捷（持续改进）
```

#### 策略 3：场景混合

```
新功能探索 → Vibe Coding
确定方案后 → TDD 重写
团队协作 → 敏捷框架
复杂逻辑 → 结对编程
日常开发 → Vibe Coding
```

### 6.4 实际建议

#### 对于个人开发者

**推荐组合**：
```
日常开发：Vibe Coding（80%）
核心功能：TDD（20%）
```

**理由**：
- 最大化个人生产力
- 关键部分保证质量

#### 对于小团队（2-5 人）

**推荐组合**：
```
项目管理：轻量级敏捷（Scrum 简化版）
功能开发：Vibe Coding
复杂功能：结对编程
核心逻辑：TDD
```

**理由**：
- 保持灵活性
- 平衡速度和质量
- 知识共享

#### 对于中大型团队（5+ 人）

**推荐组合**：
```
项目管理：标准敏捷（Scrum）
架构设计：传统方法（详细设计）
功能开发：敏捷 + Vibe Coding
关键系统：TDD
知识传递：结对编程
```

**理由**：
- 需要明确的流程
- 需要质量保证
- 需要团队协作

## 7. 总结

### 7.1 核心要点

**没有"最好"的方法，只有"最合适"的方法**

每种方法都有其适用场景：
- **瀑布**：需求明确、合规要求
- **敏捷**：团队协作、持续交付
- **TDD**：质量要求、复杂逻辑
- **结对**：知识共享、实时审查
- **Vibe Coding**：快速验证、个人项目

### 7.2 Vibe Coding 的定位

Vibe Coding 不是要取代其他方法，而是：

✅ **补充现有方法**
- 在敏捷框架内使用
- 与 TDD 结合使用
- 与结对编程结合使用

✅ **扩展应用场景**
- 让非专业程序员也能编程
- 让个人开发者更高效
- 让探索性编程更容易

✅ **适应 AI 时代**
- 充分利用 AI 能力
- 改变人机协作方式
- 提升整体生产力

### 7.3 选择建议

**问自己这些问题**：

1. **项目风险有多高？**
   - 高 → 传统方法
   - 低 → Vibe Coding

2. **需求是否明确？**
   - 明确 → 瀑布/敏捷
   - 模糊 → Vibe Coding

3. **是否团队协作？**
   - 是 → 敏捷
   - 否 → Vibe Coding

4. **质量要求如何？**
   - 极高 → TDD
   - 一般 → Vibe Coding

5. **时间是否紧迫？**
   - 是 → Vibe Coding
   - 否 → 其他方法

**最终建议**：
- 从 Vibe Coding 开始（快速验证）
- 根据需要引入其他方法（质量提升）
- 灵活组合，不要教条
