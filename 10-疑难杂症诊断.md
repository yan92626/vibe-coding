## 1. 决策树式的问题诊断流程

### 诊断起点：你遇到了什么问题？

使用这个决策树快速定位问题类型：

```
开始诊断
├─ AI 产出不符合预期？
│  ├─ 是 → 转到 A（需求沟通问题）
│  └─ 否 → 继续
├─ 效率低下，进展缓慢？
│  ├─ 是 → 转到 B（工作流问题）
│  └─ 否 → 继续
├─ 代码质量有问题？
│  ├─ 是 → 转到 C（质量控制问题）
│  └─ 否 → 继续
└─ 不知道问题在哪？
   └─ 是 → 转到 D（系统诊断）
```

### 分支 A：AI 产出不符合预期

**诊断问题**：

```
AI 产出不符合预期
├─ AI 理解错了需求？
│  ├─ 是 → 问题：需求描述不清
│  │  解决：使用需求模板，提供更多上下文
│  └─ 否 → 继续
├─ AI 用了错误的技术？
│  ├─ 是 → 问题：约束条件不明确
│  │  解决：明确技术栈和约束
│  └─ 否 → 继续
├─ AI 生成的代码有错？
│  ├─ 是 → 问题：没有验证就接受
│  │  解决：运行测试，审查代码
│  └─ 否 → 继续
└─ AI 过度设计了？
   └─ 是 → 问题：没有强调简单
      解决：明确要求"保持简单"
```

**快速修复**：
1. 停止继续，先修复当前问题
2. 重新描述需求，更清晰具体
3. 明确约束条件和技术要求
4. 让 AI 重新生成
5. 验证新的产出

**预防措施**：
- 使用需求描述模板
- 明确技术约束
- 提供示例和参考
- 确认理解后再实现

### 分支 B：效率低下，进展缓慢

**诊断问题**：

```
效率低下
├─ 花很多时间在返工？
│  ├─ 是 → 问题：没有验证就继续
│  │  解决：每步都验证
│  └─ 否 → 继续
├─ 陷入细节无法自拔？
│  ├─ 是 → 问题：追求完美
│  │  解决：设置时间限制，"足够好"就继续
│  └─ 否 → 继续
├─ 频繁切换方向？
│  ├─ 是 → 问题：决策不坚定
│  │  解决：快速决策，给方案足够时间
│  └─ 否 → 继续
├─ 每次都从零开始？
│  ├─ 是 → 问题：没有模式库
│  │  解决：建立个人模式库
│  └─ 否 → 继续
└─ 等待 AI 时间长？
   └─ 是 → 问题：工具选择不当
      解决：换更快的工具或并行工作
```

**快速修复**：
1. 记录最近一周的时间分配
2. 找出最耗时的环节
3. 针对性优化
4. 建立高效工作流

**预防措施**：
- 小步迭代，频繁验证
- 设置时间盒，避免过度优化
- 建立模式库，提高复用
- 选择合适的工具

### 分支 C：代码质量有问题

**诊断问题**：

```
代码质量差
├─ 代码难以理解？
│  ├─ 是 → 问题：没有重构
│  │  解决：定期重构，改进结构
│  └─ 否 → 继续
├─ 有很多重复代码？
│  ├─ 是 → 问题：没有提取公共逻辑
│  │  解决：提取函数，减少重复
│  └─ 否 → 继续
├─ 经常出 bug？
│  ├─ 是 → 问题：缺少测试和验证
│  │  解决：添加测试，严格验证
│  └─ 否 → 继续
├─ 有安全漏洞？
│  ├─ 是 → 问题：没有安全审查
│  │  解决：安全审查，使用安全库
│  └─ 否 → 继续
└─ 性能很差？
   └─ 是 → 问题：没有性能优化
      解决：性能分析，针对性优化
```

**快速修复**：
1. 审查当前代码
2. 列出主要问题
3. 按优先级修复
4. 建立质量标准

**预防措施**：
- 明确质量要求
- 定期代码审查
- 添加自动化测试
- 使用 Linter 和 Formatter

### 分支 D：不知道问题在哪

**系统诊断流程**：

```
第 1 步：收集信息
- 最近做了什么？
- 什么时候开始感觉不对？
- 具体表现是什么？
- 有什么异常？

第 2 步：对照检查清单
- 需求描述清晰吗？
- 工作流程合理吗？
- 代码质量如何？
- 效率如何？

第 3 步：寻找模式
- 是否重复出现？
- 什么情况下出现？
- 有什么共同点？

第 4 步：假设验证
- 提出假设
- 设计验证方法
- 验证假设
- 确定根因

第 5 步：制定方案
- 如何解决？
- 需要多久？
- 如何验证？
```

**系统检查清单**：
```
□ 需求层面
  - 需求清晰吗？
  - 目标明确吗？
  - 约束清楚吗？

□ 沟通层面
  - AI 理解对了吗？
  - 表达清楚了吗？
  - 有误解吗？

□ 执行层面
  - 流程合理吗？
  - 验证充分吗？
  - 质量达标吗？

□ 工具层面
  - 工具合适吗？
  - 用对了吗？
  - 需要换吗？

□ 心态层面
  - 心态对吗？
  - 有压力吗？
  - 需要调整吗？
```

## 3. 不同症状的深层原因分析

### 症状 1：花了很多时间但进展很少

**表面现象**：
- 每天都在忙，但看不到成果
- 感觉很努力，但项目没进展
- 时间都不知道花哪里了

**可能的深层原因**：

**原因 1：陷入细节**
- 在不重要的地方花太多时间
- 追求完美，无法完成
- 失去大局观

**原因 2：频繁返工**
- 没有验证就继续
- 方向错了，推倒重来
- 基础不牢，反复修改

**原因 3：决策瘫痪**
- 不知道选哪个方案
- 频繁切换方向
- 永远在开始

**诊断方法**：
```
记录一周的时间分配：
- 每天记录做了什么
- 每项任务花了多久
- 分析时间都去哪了

找出时间黑洞：
- 哪里花时间最多？
- 是否值得？
- 如何优化？
```

**解决路径**：
1. 设置时间盒，限制每个任务的时间
2. 小步迭代，频繁验证
3. 快速决策，给方案足够时间
4. 定期回顾，及时调整

---

### 症状 2：代码越来越乱，难以维护

**表面现象**：
- 代码看不懂，不知道在做什么
- 不敢修改，怕改坏
- 添加新功能越来越难

**可能的深层原因**：

**原因 1：只关注功能，忽视质量**
- 能跑就行，不重构
- 技术债累积
- 代码腐化

**原因 2：没有设计，随意添加**
- 想到哪写到哪
- 没有整体规划
- 结构混乱

**原因 3：复制粘贴，不提取公共逻辑**
- 重复代码多
- 修改一处要改多处
- 容易遗漏

**诊断方法**：
```
代码质量评估：
- 能快速理解吗？
- 有重复代码吗？
- 结构合理吗？
- 容易修改吗？

技术债评估：
- 有哪些技术债？
- 影响有多大？
- 需要还吗？
```

**解决路径**：
1. 停止添加新功能
2. 先重构整理代码
3. 提取公共逻辑
4. 改进结构
5. 建立质量标准
6. 定期重构

---

### 症状 3：AI 总是理解错我的需求

**表面现象**：
- AI 生成的代码不是我想要的
- 反复解释，AI 还是不懂
- 浪费很多时间沟通

**可能的深层原因**：

**原因 1：需求描述不清**
- 太模糊，AI 无法理解
- 缺少上下文
- 没有示例

**原因 2：表达方式有问题**
- 用词不准确
- 逻辑不清晰
- 信息不完整

**原因 3：期望不合理**
- 期望 AI 猜测你的想法
- 期望 AI 理解业务逻辑
- 期望一次就完美

**诊断方法**：
```
需求描述检查：
- 我说清楚了吗？
- 有歧义吗？
- 上下文够吗？
- 有示例吗？

沟通方式检查：
- 我确认理解了吗？
- 我提供了足够信息吗？
- 我的期望合理吗？
```

**解决路径**：
1. 使用需求描述模板
2. 提供更多上下文和示例
3. 确认理解后再实现
4. 迭代澄清需求
5. 调整期望，接受不完美

## 总结

**诊断流程**：
1. 识别症状
2. 使用决策树定位问题类型
3. 对照检查清单找出具体问题
4. 分析深层原因
5. 制定解决方案
6. 执行并验证

**关键原则**：
- 系统化诊断，不要盲目修复
- 找到根因，不要只治标
- 小步修复，持续验证
- 建立预防机制，避免重复
