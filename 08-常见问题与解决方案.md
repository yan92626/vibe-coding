## A. 需求描述问题

### 问题 A1：不知道如何描述需求

#### 问题描述

你有一个想法，但不知道如何用语言描述给 AI：
- "我想做个东西，但不知道怎么说"
- "我脑子里有画面，但说不清楚"
- "我不知道该从哪里开始描述"

#### 根本原因

1. **想法本身不清晰**
   - 只有模糊的感觉
   - 没有具体的目标
   - 不知道要什么

2. **缺少描述框架**
   - 不知道该说什么
   - 不知道该说多详细
   - 不知道什么信息重要

3. **担心描述不专业**
   - 怕用词不准确
   - 怕说得不够技术
   - 怕 AI 听不懂

#### 解决方案

**方案 1：用"两步法"描述需求**

第一步：描述你想要的**结果**
```
不要说："我想用 React 做一个..."
而是说："我想要一个能让用户..."

例子：
"我想要一个页面，用户可以看到商品列表，
点击商品能看到详情，可以加入购物车"
```

第二步：让 AI 帮你补充细节
```
你："我想要一个任务管理的东西"

AI："你说的任务管理，具体是指：
    1. 个人待办事项？
    2. 团队项目管理？
    3. 还是其他？"

你："个人待办事项"

AI："明白了，通常包括：
    - 添加任务
    - 标记完成
    - 删除任务
    - 设置优先级
    你需要哪些功能？"

你："前三个就够了"
```

**方案 2：用类比描述**

```
"我想做一个类似于 [已知产品] 的东西，但是..."

例子：
"我想做一个类似于 Trello 的任务管理，
但是更简单，只要基础的列表功能"

"我想做一个类似于 Instagram 的照片分享，
但是只在公司内部用"
```

**方案 3：用场景描述**

```
描述用户使用场景：

"用户打开页面，看到一个输入框，
输入任务名称，点击添加，
任务出现在下面的列表中，
点击任务可以标记完成，
完成的任务有删除线"
```

**方案 4：用问题描述**

```
描述你要解决的问题：

"我每天有很多事情要做，经常忘记，
我需要一个地方记录这些事情，
完成后能划掉，这样就不会忘了"
```

#### 实际案例

**案例 1：从模糊到清晰**

❌ **模糊描述**：
```
用户："帮我做个网站"
AI："什么类型的网站？"
用户："就是一个网站"
AI："..."
```

✅ **清晰描述**：
```
用户："我想做一个展示我的作品的网站，
      访客可以看到我的项目列表，
      点击项目能看到详细介绍和截图"

AI："明白了，这是一个作品集网站。
    需要这些功能：
    1. 首页展示项目列表
    2. 项目详情页
    3. 关于我页面
    对吗？"

用户："对，就是这样"
```

**案例 2：用类比**

```
用户："我想做一个类似于知乎的问答社区，
      但是只在我们公司内部用，
      不需要那么复杂的功能"

AI："明白了，你需要：
    - 发布问题
    - 回答问题
    - 点赞/投票
    - 搜索问题
    对吗？"

用户："对，但是不需要投票，只要能回答就行"
```

#### 预防措施

**建立你的需求描述模板**：

```
我想做：[一句话描述]

目标用户：[谁会用]
使用场景：[什么时候用，怎么用]
核心功能：[最重要的 2-3 个功能]
不需要：[明确不要什么]

例子：
我想做：个人任务管理工具

目标用户：我自己
使用场景：每天早上列出今天要做的事，
          完成后划掉
核心功能：
  - 添加任务
  - 标记完成
  - 删除任务
不需要：团队协作、提醒、分类等复杂功能
```

### 问题 A2：描述了但 AI 理解错误

#### 问题描述

你觉得自己说得很清楚了，但 AI 生成的东西完全不是你想要的：
- "我明明说了要 A，AI 却做了 B"
- "AI 理解的和我想的完全不一样"
- "我该怎么让 AI 理解我的意思？"

#### 根本原因

1. **歧义表达**
   - 同一个词有多种理解
   - 上下文不够清晰
   - 假设 AI 知道你的背景

2. **缺少关键信息**
   - 遗漏了重要约束
   - 没说明优先级
   - 没提供示例

3. **AI 的理解偏差**
   - AI 基于常见场景理解
   - 你的场景可能特殊
   - AI 可能过度推断

#### 解决方案

**方案 1：提供具体示例**

❌ **抽象描述**：
```
"我想要一个好看的按钮"
→ AI 不知道你的"好看"是什么样
```

✅ **具体示例**：
```
"我想要一个按钮，像 GitHub 的绿色按钮那样，
圆角、有阴影、鼠标悬停时颜色变深"
→ AI 知道具体要什么样
```

**方案 2：明确约束条件**

❌ **缺少约束**：
```
"帮我实现用户登录"
→ AI 可能用复杂的 OAuth
```

✅ **明确约束**：
```
"帮我实现用户登录，要求：
- 只用用户名和密码
- 不需要第三方登录
- 不需要邮箱验证
- 简单就好"
→ AI 知道要做简单版本
```

**方案 3：确认理解**

```
你："我想做一个任务管理器"

AI：生成代码

你："等等，先不要写代码。
    请先告诉我你理解的任务管理器是什么样的？"

AI："我理解的是：
    - 可以添加任务
    - 可以设置截止日期
    - 可以分配给团队成员
    - 可以设置优先级
    ..."

你："不对，我只需要最简单的：
    添加、完成、删除任务，就这三个功能"

AI："明白了，那我重新设计..."
```

**方案 4：迭代澄清**

```
第一轮：
你："我想要一个表格"
AI：生成一个数据表格

你："不对，我要的是表单，不是表格"
AI：生成一个表单

你："对，但是我要横向布局，不是竖向"
AI：调整为横向布局

你："好了，就是这样"
```

#### 实际案例

**案例 1：歧义导致的误解**

❌ **有歧义**：
```
用户："我想要一个卡片"

AI 理解：信用卡？名片？UI 卡片组件？
→ 生成了 UI 卡片组件

用户："不对，我要的是扑克牌"
```

✅ **消除歧义**：
```
用户："我想要一个扑克牌的卡片，
      显示花色和点数，
      像真实扑克牌那样"

AI："明白了，扑克牌卡片"
→ 生成正确的扑克牌
```

**案例 2：缺少关键信息**

❌ **信息不足**：
```
用户："帮我做一个搜索功能"

AI：生成了复杂的全文搜索
→ 包括分词、高亮、排序等

用户："太复杂了，我只要简单的过滤"
```

✅ **信息完整**：
```
用户："帮我做一个搜索功能，要求：
      - 只搜索任务标题
      - 包含关键词就显示
      - 不需要高亮
      - 不需要排序
      - 越简单越好"

AI：生成简单的过滤功能
→ 符合预期
```

#### 预防措施

**使用"确认式"对话**：

```
步骤 1：描述需求
你："我想做 X"

步骤 2：要求 AI 复述
你："请告诉我你理解的 X 是什么样的"

步骤 3：确认或纠正
你："对，就是这样" 或 "不对，应该是..."

步骤 4：开始实现
你："好，现在请实现"
```

**建立"需求检查清单"**：

```
描述需求时，检查是否说明了：
□ 具体要做什么？
□ 不要做什么？
□ 有什么约束？
□ 优先级如何？
□ 有没有示例？
```

### 问题 A3：需求太模糊，AI 产出不符合预期

#### 问题描述

你的需求本身就很模糊，不知道具体要什么，导致 AI 的产出总是不对：
- "我也不知道要什么，做了才知道"
- "我有个大概的想法，但说不清细节"
- "AI 做的都不是我想要的"

#### 根本原因

1. **需求本身不明确**
   - 你自己也不清楚要什么
   - 只有模糊的方向
   - 需要探索才能明确

2. **期望 AI 猜测**
   - 期望 AI 理解你的想法
   - 期望 AI 做出"完美"的东西
   - 没有给 AI 足够的信息

3. **缺少迭代过程**
   - 期望一次到位
   - 没有通过反馈逐步明确
   - 不接受"不完美"的中间版本

#### 解决方案

**方案 1：用"探索式"对话**

不要直接要求 AI 实现，而是先探索需求：

```
第一步：描述大方向
你："我想做一个帮助我管理时间的工具"

第二步：让 AI 提问
AI："时间管理工具有很多类型：
    1. 待办事项列表
    2. 番茄钟计时器
    3. 时间追踪记录
    4. 日程安排
    你想要哪种？"

第三步：逐步明确
你："我想要的是记录我每天做了什么，
    花了多少时间"

AI："明白了，是时间追踪。
    需要这些功能：
    - 开始/停止计时
    - 记录活动名称
    - 查看历史记录
    对吗？"

你："对，就是这样"
```

**方案 2：用"原型迭代法"**

先做一个最简单的版本，看了再说：

```
你："我想做一个任务管理的东西，
    但我不确定具体要什么功能"

AI："那我们先做一个最简单的版本：
    - 输入框添加任务
    - 列表显示任务
    - 点击删除任务
    你看看这个方向对不对"

AI：生成最简版本

你：使用后发现
    → "需要能标记完成"
    → "需要能编辑任务"
    → "需要能排序"

逐步添加功能，最终明确需求
```

**方案 3：用"参考对比法"**

找类似的产品作为参考：

```
你："我想做一个笔记应用，
    但不知道具体要什么功能"

AI："可以参考这些笔记应用：
    1. 简单型：Apple Notes（基础编辑）
    2. 功能型：Notion（数据库、协作）
    3. 专业型：Obsidian（双链、知识图谱）
    你想要哪种类型？"

你："我想要简单型的，像 Apple Notes"

AI："明白了，那就做基础的：
    - 创建笔记
    - 编辑内容
    - 删除笔记
    - 搜索笔记"
```

**方案 4：用"场景驱动法"**

描述具体使用场景，从场景推导需求：

```
你："我不知道要做什么，
    但我每天都会遇到这个问题：
    
    早上起来，不知道今天要做什么，
    经常忘记重要的事情，
    做完了也不知道今天完成了什么"

AI："根据你的场景，你需要：
    1. 早上能看到今天的任务（任务列表）
    2. 不会忘记（提醒功能）
    3. 知道完成了什么（完成记录）
    
    我们先做第 1 个，任务列表，
    其他的以后再加"

你："好，就这样"
```

#### 实际案例

**案例：从模糊到清晰的完整过程**

```
第 1 轮：模糊需求
你："我想做一个工具"
AI："什么类型的工具？"
你："帮助我工作的工具"
AI："具体是什么工作？"

第 2 轮：缩小范围
你："我是设计师，经常需要管理很多设计文件"
AI："你是想做：
    1. 文件管理器？
    2. 设计资源库？
    3. 项目管理工具？"
你："设计资源库"

第 3 轮：明确功能
AI："设计资源库通常包括：
    - 上传设计文件
    - 分类管理
    - 搜索查找
    - 预览查看
    你需要哪些？"
你："上传、分类、搜索就够了"

第 4 轮：确定细节
AI："好的，具体来说：
    - 支持什么格式？（PNG、JPG、SVG...）
    - 如何分类？（文件夹、标签...）
    - 搜索什么？（文件名、标签...）"
你："PNG 和 JPG，用文件夹分类，搜索文件名"

第 5 轮：开始实现
AI："明白了，现在开始实现..."
```

#### 预防措施

**接受"需求不明确"是正常的**：

```
不要期望：一开始就知道所有细节
而是接受：需求是逐步明确的过程

方法：
1. 先做最小版本
2. 使用后发现问题
3. 逐步添加功能
4. 最终明确需求
```

**建立"探索式"工作流**：

```
阶段 1：探索（不写代码）
- 和 AI 讨论可能的方向
- 了解不同的选择
- 明确大致方向

阶段 2：原型（快速实现）
- 做最简单的版本
- 能跑就行
- 快速验证

阶段 3：反馈（使用测试）
- 实际使用
- 发现问题
- 明确需求

阶段 4：迭代（逐步完善）
- 根据反馈调整
- 添加必要功能
- 优化体验
```

## B. 技术知识问题

### 问题 B1：不懂技术细节，无法给出约束

#### 问题描述

这是我们在对话开始时讨论的核心问题：
- "我不知道该用什么技术"
- "我不懂这些技术术语"
- "我无法给 AI 具体的技术要求"
- "AI 用了我不理解的技术"

#### 根本原因

1. **知识差距**
   - 不了解技术选项
   - 不知道各种方案的优劣
   - 缺少技术背景

2. **期望错误**
   - 认为必须懂技术才能用 Vibe Coding
   - 认为必须给出技术约束
   - 不知道可以让 AI 帮助学习

3. **沟通方式错误**
   - 直接要求实现
   - 不先了解技术方案
   - 跳过学习环节

#### 解决方案

**方案 1：两步法（我们之前讨论的核心方法）**

**第一步：让 AI 当"技术顾问"（不写代码）**

```
你："我想做用户登录，但我不懂技术细节。

请告诉我：
1. 需要考虑哪些关键问题？
2. 有哪些常见方案？
3. 每个方案的优缺点是什么？
4. 对于我的场景（个人项目，快速验证），
   你推荐哪种？"

AI："用户登录需要考虑：

关键问题：
1. 如何存储密码？（加密方式）
2. 如何保持登录状态？（Session/JWT）
3. 如何防止攻击？（SQL注入、XSS）

常见方案：
A. Session + Cookie
   优点：简单，服务器控制
   缺点：需要服务器存储状态
   
B. JWT Token
   优点：无状态，易扩展
   缺点：无法主动失效
   
C. OAuth 第三方登录
   优点：安全，用户方便
   缺点：依赖第三方

推荐：
对于个人项目快速验证，推荐方案 A（Session）
原因：实现简单，够用"

你："明白了，那就用 Session 方案"
```

**第二步：基于理解后再要代码**

```
你："好，现在按照 Session 方案实现登录，要求：
    - 密码用 bcrypt 加密
    - Session 存储在服务器
    - 登录后跳转到首页"

AI：生成代码

你：因为理解了方案，所以能判断代码对不对
```

**方案 2：边做边学法**

不要等"学会了"再做，而是边做边学：

```
你："我想做一个数据展示页面，
    但我不知道该用什么图表库"

AI："常用的图表库有：
    1. Chart.js - 简单易用
    2. ECharts - 功能强大
    3. D3.js - 高度定制
    
    对于初学者，推荐 Chart.js"

你："为什么推荐 Chart.js？"

AI："因为：
    1. API 简单，容易上手
    2. 文档清晰
    3. 满足大部分需求
    4. 不需要深入学习"

你："好，那就用 Chart.js。
    请给我一个简单的例子"

AI：生成示例代码

你：运行，看效果，理解用法

你："明白了，现在帮我实现我的数据展示"
```

**方案 3：问"为什么"**

遇到不懂的技术，问 AI 解释：

```
AI："建议使用 Redux 管理状态"

你："等等，什么是 Redux？为什么需要它？"

AI："Redux 是状态管理库。

什么时候需要：
- 多个组件共享状态
- 状态逻辑复杂
- 需要时间旅行调试

你的场景：
- 只有简单的任务列表
- 状态不复杂

结论：你不需要 Redux，用 useState 就够了"

你："明白了，那就用 useState"
```

**方案 4：用"约束"而非"技术"描述需求**

不要用技术术语，用约束条件描述：

❌ **技术描述**：
```
"用 React Hooks 的 useReducer 和 Context API
实现全局状态管理"
→ 需要懂这些技术
```

✅ **约束描述**：
```
"我有多个页面需要共享用户信息，
希望实现方式简单，容易理解"
→ AI 会选择合适的技术
```

#### 实际案例

**案例 1：从不懂到理解**

```
第 1 轮：暴露不懂
你："我想做一个网站，但我不知道该用什么技术"

AI："网站开发有几种选择：
    1. 纯静态（HTML+CSS+JS）
    2. 前端框架（React/Vue）
    3. 全栈框架（Next.js）
    
    你的网站需要：
    - 动态内容吗？
    - 用户交互多吗？
    - 需要 SEO 吗？"

第 2 轮：提供信息
你："只是展示我的作品，不需要用户登录，
    希望 Google 能搜到"

AI："那推荐：
    - 纯静态网站（简单）
    - 或 Next.js（更好的 SEO）
    
    如果追求简单，用纯静态
    如果要更好的 SEO 和未来扩展性，用 Next.js"

第 3 轮：做决定
你："我选简单的，纯静态"

```javascript
// O(n²) 复杂度
function findDuplicates(arr) {
  const duplicates = [];
  for (let i = 0; i < arr.length; i++) {
    for (let j = i + 1; j < arr.length; j++) {
      if (arr[i] === arr[j]) {
        duplicates.push(arr[i]);
      }
    }
  }
  return duplicates;
}
```

你："数据量大时很慢，如何优化？"

```javascript
// O(n) 复杂度
function findDuplicates(arr) {
  const seen = new Set();
  const duplicates = new Set();
  
  for (const item of arr) {
    if (seen.has(item)) {
      duplicates.add(item);
    }
    seen.add(item);
  }
  
  return Array.from(duplicates);
}

→ 停止当前任务
→ 回到重要的事情
→ 记录细节问题，以后再处理
```

**方案 4：使用"够用就好"原则**

```
问自己：
1. 当前状态够用吗？
2. 继续优化的收益大吗？
3. 时间可以用在更重要的地方吗？

如果答案是"够用"、"不大"、"是"：
→ 停止优化
→ 转向更重要的任务
```

#### 实际案例

**案例：从陷入到跳出**

```
场景：做任务管理应用

陷入细节：
第 1 天：调整按钮颜色（2 小时）
第 2 天：继续调整按钮颜色（3 小时）
第 3 天：还在调整按钮颜色（2 小时）
→ 花了 7 小时在按钮颜色上
→ 其他重要功能都没做

跳出来：
"等等，我在干什么？
按钮颜色真的这么重要吗？
我还有任务编辑、删除功能没做呢！"

调整：
1. 按钮颜色"够用就好"
2. 转向重要功能
3. 设置时间限制：
   - 任务编辑：2 小时
   - 任务删除：1 小时
   - 不追求完美，能用就行
```

#### 预防措施

**建立"大局观"习惯**：

```
每天开始：
□ 列出今天要做的 3 件最重要的事
□ 优先做这 3 件事
□ 其他的"够用就好"

每 2 小时：
□ 停下来，回顾一下
□ 我在做什么？
□ 这个重要吗？
□ 是否陷入细节？

每天结束：
□ 今天完成了重要的事吗？
□ 有没有浪费时间在细节上？
□ 明天应该做什么？
```
