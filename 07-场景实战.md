## 1. 快速原型开发

### 场景特点

```
目标：快速验证想法
时间：1-3 天
质量要求：能跑就行
重点：速度 > 质量
```

### 推荐策略

#### 策略 1：最小功能集

```
只做核心功能：
必须有的功能
可选的功能
未来可能要的功能

例子：任务管理原型
添加任务
显示任务
编辑任务（以后再说）
任务分类（以后再说）
提醒功能（以后再说）
```

#### 策略 2：使用最简单的技术

```
不要：
- 复杂的技术栈
- 需要学习的新技术
- 过度设计

而是：
- 你熟悉的技术
- 最简单的方案
- 能快速实现的

例子：
不用：React + Redux + TypeScript
而用：纯 HTML + JavaScript
```

#### 策略 3：忽略边界情况

```
原型阶段可以忽略：
- 错误处理（假设一切正常）
- 输入验证（假设输入正确）
- 性能优化（数据量小）
- 安全性（只是原型）

专注核心流程
```

### 完整对话示例

```
第 1 轮：明确目标
你："我想快速验证一个想法：
    用户可以记录每天的心情，
    看到心情的变化趋势。
    
    这是原型，只要能演示核心功能，
    3 天内完成"

AI："明白，原型开发。建议：
    - 用纯 HTML + JavaScript（最简单）
    - 只做：记录心情、显示列表、简单图表
    - 不做：用户系统、数据导出、复杂分析
    对吗？"

你："对，就这样"

第 2 轮：快速实现
你："先做最简单的版本：
    - 一个输入框选择心情（开心/一般/难过）
    - 一个按钮记录
    - 显示历史记录
    - 用 localStorage 存储"

AI：生成代码

你：运行，测试
    → 能用 ✓

第 3 轮：添加可视化
你："加一个简单的图表，
    显示最近 7 天的心情趋势"

AI：添加图表

你：运行，测试
    → 能看到趋势 ✓

第 4 轮：完成
你："够了，原型完成"
    → 总用时：4 小时
    → 成功验证想法
```

### 注意事项

```
做：
- 快速迭代
- 频繁测试
- 专注核心
- 接受不完美

不做：
- 追求完美
- 过度设计
- 添加非核心功能
- 优化性能
```

## 2. 学习新技术栈

### 场景特点

```
目标：学习并掌握新技术
时间：1-2 周
质量要求：理解原理
重点：学习 > 产出
```

### 推荐策略

#### 策略 1：从最简单的例子开始

```
第 1 步：Hello World
你："给我一个最简单的 [技术] 例子"
AI：生成 Hello World
你：运行，理解基本结构

第 2 步：加一点功能
你："加一个按钮，点击显示文字"
AI：添加交互
你：运行，理解事件处理

第 3 步：加状态管理
你："让按钮点击次数显示出来"
AI：添加状态
你：运行，理解状态管理

逐步深入
```

#### 策略 2：边做边问

```
不要：
- 先学完所有理论
- 看完所有文档
- 然后才开始做

而是：
- 先做最简单的
- 遇到不懂的就问
- 边做边学

例子：
你："这个 useEffect 是什么？"
AI：解释 useEffect
你："什么时候用？"
AI：说明使用场景
你："给个例子"
AI：提供例子
你：理解了，继续
```

#### 策略 3：做一个完整项目

```
学习新技术的最好方式：
做一个小而完整的项目

项目特点：
- 包含主要概念
- 不太复杂
- 有实际价值
- 能完成

例子：学习 React
做一个任务管理应用：
- 组件（Component）
- 状态（State）
- 事件处理（Event）
- 列表渲染（List）
- 条件渲染（Conditional）

通过项目掌握核心概念
```

```
做：
- 从简单开始

不做：
- 试图一次学完
- 只看不做
- 跳过基础
- 追求高级技巧

## 3. 调试和修复 Bug

### 场景特点

```
目标：找到并修复问题
时间：不确定
质量要求：彻底解决
重点：定位 > 修复
```

### 推荐策略

#### 策略 1：系统化定位

```
第 1 步：复现问题
- 确定触发条件
- 记录错误信息
- 确认影响范围

第 2 步：缩小范围
- 哪个模块？
- 哪个函数？
- 哪一行？

第 3 步：分析原因
- 为什么出错？
- 什么情况下出错？
- 预期是什么？

第 4 步：验证假设
- 添加日志
- 断点调试
- 测试假设

第 5 步：修复验证
- 修复问题
- 测试修复
- 确认解决
```

#### 策略 2：让 AI 协助诊断

```
你："我遇到一个 bug：
    [描述问题]
    
    错误信息：
    [粘贴错误]
    
    相关代码：
    [粘贴代码]
    
    请帮我分析可能的原因"

AI：分析问题，提供可能原因

你："我试了 [方法]，还是不行"

AI：提供其他可能性

你：逐个排查，找到原因
```

#### 策略 3：橡皮鸭调试法

```
向 AI 详细描述问题：

你："我遇到一个奇怪的问题：
    当用户点击按钮时，
    应该显示弹窗，
    但是什么都没发生。
    
    我检查了：
    1. 按钮的事件绑定 - 正常
    2. 函数被调用了 - 确认
    3. 弹窗组件存在 - 确认
    
    但就是不显示..."

在描述过程中，你可能自己发现问题
或者 AI 能指出你忽略的地方
```

### 完整对话示例

```
场景：登录功能不工作

第 1 轮：描述问题
你："登录功能有问题：
    点击登录按钮后，
    页面没有任何反应，
    也没有错误提示"

AI："请提供：
    1. 浏览器控制台的错误信息
    2. 登录相关的代码
    3. 网络请求是否发送"

第 2 轮：提供信息
你："控制台错误：
    TypeError: Cannot read property 'value' of null
    
    代码：
    [粘贴登录代码]
    
    网络请求：没有发送"

AI："问题找到了：
    document.getElementById('username')
    返回 null，说明元素不存在。
    
    可能原因：
    1. ID 拼写错误
    2. 脚本在 DOM 加载前执行
    3. 元素在其他地方"

第 3 轮：验证修复
你："检查了，ID 是对的，
    但脚本确实在 DOM 前执行了"

AI："解决方案：
    1. 把脚本放在 body 底部
    2. 或使用 DOMContentLoaded
    3. 或使用 defer 属性"

你："用了 DOMContentLoaded，
    现在可以了！"
```

### 注意事项

```
做：
- 系统化定位
- 记录调试过程
- 验证修复
- 理解根本原因

不做：
- 随机尝试
- 不理解就修改
- 修复症状不修复原因
- 不测试就认为解决了
```

## 4. 重构现有代码

### 场景特点

```
目标：改进代码质量
时间：根据范围
质量要求：保持功能不变
重点：结构 > 功能
```

### 推荐策略

#### 策略 1：小步重构

```
不要：一次重构整个文件
而是：每次只重构一小部分

步骤：
1. 提取重复代码
2. 改进命名
3. 拆分大函数
4. 优化结构
5. 添加注释

每一步都：
- 保持功能不变
- 立即测试
- 确认 OK
```

#### 策略 2：让 AI 提供建议

```
你："请审查这段代码，
    指出可以改进的地方：
    
    [粘贴代码]"

AI："建议改进：
    1. 提取重复的验证逻辑
    2. 函数名不够清晰
    3. 嵌套太深，可以提前返回
    4. 缺少错误处理"

你："好，我们一个一个来，
    先提取重复逻辑"

AI：重构代码

你：测试，确认 OK

你："继续改进命名"
...
```

#### 策略 3：重构模式

```
模式 1：提取函数
将重复代码提取为函数

模式 2：提取变量
将复杂表达式提取为变量

模式 3：简化条件
减少嵌套，提前返回

模式 4：统一命名
使用一致的命名规范

模式 5：添加类型
添加类型注解或 JSDoc
```

### 完整对话示例

```
场景：重构混乱的表单验证代码

原始代码（混乱）：
```javascript
function check(d) {
  if(d.n) {
    if(d.n.length >= 4) {
      if(d.n.length <= 20) {
        if(/^[a-zA-Z0-9]+$/.test(d.n)) {
          if(d.p) {
            if(d.p.length >= 8) {
              return true;
            }
          }
        }
      }
    }
  }
  return false;
}
```

第 1 轮：改进命名
你："这段代码很难理解，
    先改进命名"

AI：重构
```javascript
function validateForm(data) {
  if(data.username) {
    if(data.username.length >= 4) {
      if(data.username.length <= 20) {
        if(/^[a-zA-Z0-9]+$/.test(data.username)) {
          if(data.password) {
            if(data.password.length >= 8) {
              return true;
            }
          }
        }
      }
    }
  }
  return false;
}
```

你：测试，OK

第 2 轮：简化嵌套
你："嵌套太深，简化一下"

AI：重构
```javascript
function validateForm(data) {
  if (!data.username) return false;
  if (data.username.length < 4) return false;
  if (data.username.length > 20) return false;
  if (!/^[a-zA-Z0-9]+$/.test(data.username)) return false;
  if (!data.password) return false;
  if (data.password.length < 8) return false;
  return true;
}
```

你：测试，OK

第 3 轮：提取验证逻辑
你："提取验证逻辑为独立函数"

AI：重构
```javascript
function isValidUsername(username) {
  if (!username) return false;
  if (username.length < 4 || username.length > 20) return false;
  return /^[a-zA-Z0-9]+$/.test(username);
}

function isValidPassword(password) {
  return password && password.length >= 8;
}

function validateForm(data) {
  return isValidUsername(data.username) && 
         isValidPassword(data.password);
}
```

你：测试，OK，完成！

结果：代码清晰易懂
```

### 注意事项

```
做：
- 小步重构
- 每步测试
- 保持功能不变
- 理解每个改动

不做：
- 大规模重写
- 同时改功能
- 不测试就继续
- 盲目接受建议
```

## 5. 编写文档和注释

### 场景特点

```
目标：让代码易于理解
时间：根据代码量
质量要求：清晰准确
重点：可读性
```

### 推荐策略

#### 策略 1：让 AI 生成初稿

```
你："为这段代码添加注释：
    [粘贴代码]
    
    要求：
    - 函数添加 JSDoc
    - 复杂逻辑添加行内注释
    - 用中文"

AI：生成带注释的代码

你：审查，调整
```

#### 策略 2：文档类型

```
类型 1：代码注释
- 函数注释（JSDoc）
- 行内注释
- 类型注解

类型 2：API 文档
- 接口说明
- 参数说明
- 返回值说明
- 示例代码

类型 3：使用文档
- README
- 快速开始
- 使用指南
- 常见问题

类型 4：架构文档
- 系统架构
- 模块划分
- 数据流
- 技术决策
```

#### 策略 3：文档原则

```
原则 1：必要性
- 不要注释显而易见的代码
- 注释"为什么"而非"是什么"
- 复杂逻辑必须注释

原则 2：准确性
- 代码改了，注释也要改
- 注释要与代码一致
- 不要误导性注释

原则 3：简洁性
- 简明扼要
- 避免废话
- 重点突出
```

### 完整对话示例

```
场景：为 API 编写文档

 * @returns {Promise<Object>} 用户列表和分页信息
 */
async function getUsers(params) {
  // ...
}
```

第 2 轮：生成 API 文档
你："生成 API 文档，
    包括所有接口的说明"

AI：生成 Markdown 文档
```markdown
# API 文档

## 获取用户列表

**接口**：GET /api/users

**参数**：
- page: 页码（默认 1）

**示例**：

```

第 3 轮：生成 README
你："生成 README，
    包括项目介绍、安装、使用"

AI：生成 README.md

```

### 注意事项

```
做：
- 注释复杂逻辑
- 保持注释更新
- 提供示例
- 简洁清晰

不做：
- 注释显而易见的代码
- 过时的注释
- 误导性注释
- 冗长废话
```

## 7. 数据分析任务

### 场景特点

```
目标：分析和处理数据
时间：根据数据量
质量要求：结果准确
重点：分析 > 展示
```

### 推荐策略

#### 策略 1：先理解数据

```
第 1 步：查看数据结构
你："这是我的数据：
    [粘贴样本数据]
    
    请帮我理解数据结构"

AI：分析数据结构

第 2 步：确定分析目标
你："我想知道：
    1. 用户增长趋势
    2. 最活跃的用户
    3. 使用高峰时段"

第 3 步：设计分析方案
AI：提供分析方案

第 4 步：实现分析
逐步实现各个分析
```

#### 策略 2：迭代分析

```
不要：一次做完所有分析
而是：逐步深入

第 1 轮：基础统计
- 总数、平均值、中位数

第 2 轮：趋势分析
- 时间序列、增长率

第 3 轮：深入分析
- 相关性、异常值

第 4 轮：可视化
- 图表展示
```

#### 策略 3：验证结果

```
分析结果要验证：
- 数据是否合理？
- 计算是否正确？
- 有没有异常？

方法：
- 手动验证样本
- 交叉验证
- 常识判断
```

### 完整对话示例

```
场景：分析用户行为数据

第 1 轮：理解数据
你："这是用户行为数据：
    [粘贴数据]
    
    包含：用户ID、操作类型、时间戳
    
    请帮我理解数据"

AI："数据结构：
    - 每行是一次操作
    - 时间跨度：1个月
    - 用户数：约1000人
    - 操作类型：登录、浏览、购买"

第 2 轮：基础分析
你："先做基础统计：
    - 总操作数
    - 日均操作数
    - 用户活跃度"

AI：生成分析代码

你：运行，查看结果

第 3 轮：趋势分析
你："分析每日操作趋势"

AI：生成趋势分析

你：发现周末操作数下降

第 4 轮：深入分析
你："为什么周末下降？
    分析用户类型"

AI：按用户类型分析

你：发现企业用户周末不活跃

第 5 轮：可视化
你："生成图表展示发现"

AI：生成可视化

你：完成分析报告
```

### 注意事项

```
做：
- 理解数据
- 验证结果
- 迭代分析
- 可视化展示

不做：
- 盲目分析
- 不验证结果
- 过度解读
- 忽视异常
```

## 8. 自动化脚本

### 场景特点

```
目标：自动化重复任务
时间：1-2 小时
质量要求：可靠运行
重点：实用 > 优雅
```

### 推荐策略

#### 策略 1：明确任务流程

```
第 1 步：列出手动步骤
1. 打开文件
2. 读取数据
3. 处理数据
4. 保存结果

第 2 步：转化为脚本
让 AI 将步骤转化为代码

第 3 步：测试验证
用样本数据测试
```

#### 策略 2：处理异常

```
脚本必须处理：
- 文件不存在
- 数据格式错误
- 网络错误
- 权限问题

原则：
- 失败要有提示
- 不要静默失败
- 记录错误日志
```

#### 策略 3：使其可复用

```
不要：
- 硬编码路径
- 硬编码参数
- 一次性代码

而是：
- 参数化
- 配置文件
- 可重复使用
```

### 完整对话示例

```
场景：批量处理图片

第 1 轮：描述任务
你："我需要一个脚本：
    1. 读取文件夹中的所有图片
    2. 压缩到 50% 质量
    3. 保存到新文件夹
    
    用 Python 实现"

AI：生成脚本

第 2 轮：添加错误处理
你："添加错误处理：
    - 文件夹不存在
    - 图片格式不支持
    - 保存失败"

AI：添加错误处理

第 3 轮：参数化
你："改为可配置：
    - 输入文件夹
    - 输出文件夹
    - 压缩质量"

AI：改为参数化

第 4 轮：测试
你：用样本图片测试
    → 成功 ✓

第 5 轮：添加进度显示
你："处理大量图片时，
    显示进度"

AI：添加进度条

完成：可靠的自动化脚本
```

### 注意事项

```
做：
- 明确流程
- 错误处理
- 参数化
- 测试验证

不做：
- 硬编码
- 忽视异常
- 不测试
- 过度复杂
```
