## 1. 历史视角：Agent 工程的实践与痛点

**2024年下半年：从框架到实践**

在使用 LangChain、LangGraph、CrewAI 搭建 multi-agent 系统时，遇到了新的挑战：

**RAG（检索增强生成）的痛点**：

**真实场景：给 AI 提供项目文档**

我有一个 50 页的项目文档，想让 AI 回答"用户权限是如何设计的？"

**方案 1：直接放进上下文**
- 问题：50 页文档 = 约 30K tokens，太大了
- 结果：要么截断丢失信息，要么成本太高

**方案 2：用 RAG**

第一步：切分文档
```
问题来了：怎么切？

按段落切？
- 第 23 段："权限系统采用 RBAC 模型"
- 第 24 段："具体实现见下表"
- 第 25 段：[表格]
→ 如果只检索到第 23 段，AI 看不到表格

按页切？
- 第 5 页：讲权限设计
- 第 6 页：讲权限实现
- 第 7 页：讲权限测试
→ 切太大，包含太多无关信息

按语义切？
- 需要先理解文档结构
- 需要识别章节边界
- 需要处理跨章节的引用
→ 复杂度爆炸
```

第二步：向量化
```
"权限系统采用 RBAC 模型" → [0.23, 0.45, 0.12, ...]

问题：
- 用户问"如何控制访问？"
- 文档写的是"RBAC 模型"
- 这两个向量相似度不高
→ 检索失败
```

第三步：检索
```
我问："用户权限是如何设计的？"

检索结果：
1. 第 23 段："权限系统采用 RBAC 模型"（相关度 0.85）
2. 第 67 段："用户表包含 role_id 字段"（相关度 0.78）
3. 第 102 段："权限检查在中间件中实现"（相关度 0.72）

问题：
- 这三段分散在文档各处
- 缺少上下文，AI 难以理解
- 可能漏掉重要信息（比如权限继承规则在第 45 段，但相关度只有 0.65）
```

**实际遇到的问题**：

某次我问："为什么用户 A 能访问资源 B？"

RAG 检索到：
- "权限系统采用 RBAC"
- "用户 A 是管理员"
- "资源 B 需要管理员权限"

看起来完美？但实际上：
- 文档第 78 段写了："特殊情况：周末管理员权限降级"
- 这段相关度不高，没被检索到
- AI 给出了错误答案

**记忆管理的挑战**：

**真实场景：项目记忆越来越多**

第 1 天：
```
记忆：
- 项目用 React
- 用 Tailwind CSS
- API 用 RESTful
[3 条记忆，检索很快]
```

第 30 天：
```
记忆：
- 项目用 React
- 用 Tailwind CSS
- API 用 RESTful
- 按钮用蓝色
- 表单验证用 Yup
- 错误提示用 toast
- 用户反馈：喜欢简洁风格
- 用户反馈：不喜欢动画
- 技术决策：不用 Redux（太重）
- 技术决策：用 Context（够用）
- Bug 记录：Safari 兼容性问题
- Bug 记录：移动端滚动问题
... [100+ 条记忆]
```

现在我问："写一个登录表单"

AI 要检索哪些记忆？
- "用 React"（必需）
- "用 Tailwind CSS"（必需）
- "表单验证用 Yup"（必需）
- "按钮用蓝色"（需要吗？）
- "用户喜欢简洁风格"（需要吗？）
- "Safari 兼容性问题"（需要吗？）

如果全部检索：
- 100 条记忆 = 5000 tokens
- 成本高，响应慢

如果只检索相关的：
- 怎么判断"相关"？
- "用户喜欢简洁风格"算相关吗？
- 如果漏掉了，生成的表单可能不符合用户期望

**更新记忆的困境**：

第 10 天我说："我们决定用 Redux 了"

现在有两条矛盾的记忆：
- 第 5 天："不用 Redux（太重）"
- 第 10 天："用 Redux"

AI 应该怎么办？
- 删除旧记忆？但可能需要知道"为什么改变主意"
- 保留两条？但会混淆 AI
- 标记旧记忆为"已废弃"？但怎么判断哪些记忆受影响？

**框架本身的问题**：

**真实场景：用 LangChain 搭建一个问答系统**

```python
from langchain import LLMChain, PromptTemplate
from langchain.memory import ConversationBufferMemory
from langchain.agents import initialize_agent

# 看起来很简单？
```

实际遇到的问题：

**问题 1：调试困难**
```
运行报错：
"Error in chain execution"

哪里错了？
- Prompt 模板？
- Memory 配置？
- Agent 逻辑？
- LLM 调用？

需要加一堆 print 才能定位
```

**问题 2：版本升级**
```
LangChain 0.0.200 → 0.0.300

Breaking changes：
- ConversationBufferMemory 改名了
- initialize_agent 参数变了
- PromptTemplate 格式变了

我的代码全挂了
```

**问题 3：成本失控**

我搭建了一个 3-agent 系统：
- Agent 1：理解用户问题
- Agent 2：检索相关信息
- Agent 3：生成回答

看起来很合理？

实际运行：
```
用户问一个问题：

Agent 1 调用 LLM：500 tokens
Agent 1 → Agent 2 传递消息：200 tokens
Agent 2 调用 LLM：800 tokens
Agent 2 → Agent 3 传递消息：300 tokens
Agent 3 调用 LLM：1000 tokens

总计：2800 tokens

如果直接用一个 prompt：
"根据这些信息回答问题"：1200 tokens

多花了一倍多！
```

**现代工具的简化**：

**Claude Code 的 Memory：实际体验**

我不需要管理记忆，它自动做了：

```
第 1 天我说："这个项目用 React"
→ Claude Code 记住了

第 10 天我问："帮我写个组件"
→ Claude Code 自动用 React 写，不需要我再说

第 15 天我说："改用 Vue 了"
→ Claude Code 更新记忆

第 20 天我问："为什么当时选 React？"
→ Claude Code 能回答（保留了历史）
```

我不需要：
- 手动分类记忆（长期/短期）
- 手动检索记忆
- 手动更新记忆
- 手动处理冲突

**MCP 的实际价值**：

没有 MCP 时：
```
我想让 AI 访问：
- 项目文件
- Git 历史
- 数据库
- API 文档

需要写 4 个不同的集成
每个都要处理：
- 认证
- 权限
- 错误处理
- 数据格式转换
```

有了 MCP：
```
安装 4 个 MCP server：
- filesystem-server
- git-server
- database-server
- api-docs-server

Claude Code 自动知道如何使用它们
我只需要说："查看最近的 git 提交"
```

**长上下文的实际影响**：

**2023 年（4K 上下文）**：
```
我的项目有 10 个文件，每个 500 行

想让 AI 理解整个项目？
→ 不可能，放不下

只能：
- 一次处理一个文件
- 手动告诉 AI 文件间的关系
- 反复解释项目结构
```

**2026 年（200K 上下文）**：
```
10 个文件 = 约 50K tokens

可以全部放进去！

但新问题：
AI 说："我看到你在 utils.js 第 234 行定义了一个函数..."

我：？？？（我根本不记得第 234 行是什么）

→ "迷失在中间"问题
```

**实际的最佳实践**：
```
不是"全部放进去"
而是"智能选择"：

当前任务：修改登录功能

相关文件：
- login.js（必需）
- auth.js（必需）
- user.js（可能需要）
- utils.js（可能需要）

不相关：
- dashboard.js
- settings.js
- ...

只放相关的，不是全部
```

**对 Vibe Coding 的启示**：

**1. 简单方案往往更好**

我花了 2 周搭建 RAG 系统，结果发现：
- 80% 的场景，直接把文档放进上下文就够了
- 只有文档特别大时，才真正需要 RAG

**2. 不要为了技术而技术**

Multi-agent 听起来很酷，但：
- 成本高
- 调试难
- 大多数场景，一个好的 prompt 就够了

**3. 工具会解决工具的问题**

2024 年我手动管理记忆，很痛苦
2026 年 Claude Code 自动管理，我不用操心

不要过度投入在"弥补工具不足"上
工具会进化，你的投入可能白费

**4. 理解本质，而不是追逐技术**

RAG 的本质：让 AI 访问大量信息
记忆管理的本质：让 AI 记住重要信息
Agent 协调的本质：分解复杂任务

理解本质后，你会发现：
- 很多技术是在解决同一个问题
- 新工具出现时，你能快速理解它的价值
- 你不会被技术名词吓到

## 2. 知识差距弥补法

这是我们在对话开始时深入讨论的核心策略：如何在不懂技术的情况下，仍然能高效使用 Vibe Coding。

### 2.1 用 AI 学习技术概念

#### 学习策略 1：概念优先，细节其次

```
不要：
"教我 React 的所有知识"
→ 太宽泛，学不完

而是：
"我想用 React 做任务管理，
需要理解哪些核心概念？"
→ 聚焦，够用就好
```

**AI 回答示例**：
```
核心概念：
1. 组件（Component）- 构建 UI 的积木
2. 状态（State）- 数据的存储
3. Props - 组件间传递数据
4. 事件处理 - 响应用户操作

对于任务管理，主要用到：
- useState（管理任务列表）
- 事件处理（添加、删除任务）
- 列表渲染（显示任务）

先理解这些就够了
```

#### 学习策略 2：用类比理解

```
你："什么是 React 的 state？"

AI："State 就像一个记事本：
- 你在上面写东西（设置状态）
- 内容变了，页面就更新（重新渲染）
- 每个组件有自己的记事本（独立状态）

例子：
任务列表就是一个 state
添加任务 = 在记事本上写一条
删除任务 = 从记事本上划掉一条"

你：理解了！
```

#### 学习策略 3：边做边学

```
第 1 步：看最简单的例子
你："给我一个最简单的 React 例子"
AI：生成 Hello World
你：运行，理解基本结构

第 2 步：加一点功能
你："加一个按钮，点击显示文字"
AI：添加按钮和事件
你：运行，理解事件处理

第 3 步：加状态管理
你："让按钮点击次数显示出来"
AI：添加 useState
你：运行，理解状态

第 4 步：应用到实际项目
你："现在我理解了，帮我做任务管理"
```

### 2.2 两步法：先学习，再实现

这是核心方法，我们之前详细讨论过：

#### 第一步：AI 当技术顾问（不写代码）

```
你："我想做 [功能]，但我不懂技术。

请告诉我：
1. 需要考虑哪些关键问题？
2. 有哪些常见方案？
3. 每个方案的优缺点？
4. 对于我的场景，推荐哪种？
5. 为什么推荐这个？"

AI：详细解释各种方案

你：理解后，做出决策
```

#### 第二步：基于理解后再要代码

```
你："好，我决定用 [方案]。

现在请实现，要求：
- [具体约束]
- [关键要求]
- [注意事项]"

AI：生成代码

你：因为理解了方案，能判断对错
```

#### 两步法的优势

```
你理解了技术方案
能判断 AI 的输出
遇到问题知道怎么办
学到了知识

vs 一步法（直接要代码）：
不理解代码
无法判断对错
出问题不知道怎么办
什么都没学到
```

### 2.3 如何快速建立技术直觉

#### 方法 1：多问"为什么"

```
不要只接受答案，要理解原因：

AI："用 useState 管理状态"
你："为什么用 useState？"
AI："因为状态变化时需要重新渲染"
你："什么时候不用 useState？"
AI："数据不需要触发渲染时"
你："明白了"

通过追问，建立深入理解
```

#### 方法 2：对比学习

```
你："useState 和普通变量有什么区别？"

AI："对比：
普通变量：
- 改变不会触发渲染
- 组件重新渲染时会重置
- 适合临时数据

useState：
- 改变会触发渲染
- 数据在渲染间保持
- 适合 UI 相关数据"

你：通过对比，理解差异
```

#### 方法 3：实验验证

```
你："我想验证一下 useState 的行为"

AI：生成实验代码

你：运行，观察
- 改变 state → 页面更新 ✓
- 改变普通变量 → 页面不更新 ✓

你：通过实验，建立直觉
```

#### 方法 4：总结模式

```
做了几个项目后，总结模式：

"我发现的模式：
1. 需要显示在页面上的数据 → 用 state
2. 临时计算的数据 → 用普通变量
3. 不变的配置 → 用常量
4. 需要在组件间共享 → 用 props 或 context

这是我的直觉"

通过总结，形成自己的理解
```

## 3. 复杂项目的 Vibe Coding

当项目变大变复杂时，需要更系统的策略。

### 3.1 大型项目的分解策略

#### 策略 1：垂直切片

```
不要：按技术层分解
第 1 周：做所有的数据库
第 2 周：做所有的 API
第 3 周：做所有的 UI

而是：按功能切片
第 1 周：用户登录（数据库+API+UI）
第 2 周：任务管理（数据库+API+UI）
第 3 周：数据统计（数据库+API+UI）

优势：
- 每周都有可用的功能
- 及早发现问题
- 可以调整优先级
```

#### 策略 2：MVP 优先

```
第一版：最小可用产品
- 只做核心功能
- 能用就行
- 快速验证

第二版：完善功能
- 添加重要功能
- 优化体验
- 修复问题

第三版：打磨产品
- 添加辅助功能
- 性能优化
- 完善细节

不要一开始就追求完美
```

#### 策略 3：模块化开发

```
项目结构：
project/
├── auth/          # 认证模块（独立）
├── tasks/         # 任务模块（独立）
├── users/         # 用户模块（独立）
└── shared/        # 共享代码

每个模块：
- 独立开发
- 独立测试
- 清晰接口
- 低耦合

优势：
- 可以并行开发
- 容易维护
- 容易测试
```

#### 策略 4：迭代深化

```
第 1 轮：实现基础版本
- 核心流程能跑
- 不考虑边界情况
- 不考虑性能

第 2 轮：完善功能
- 处理边界情况
- 添加错误处理
- 改进用户体验

第 3 轮：优化提升
- 性能优化
- 代码重构
- 完善测试

每一轮都是完整可用的
```

### 3.2 架构层面的 Vibe Coding

#### 架构设计流程

```
第 1 步：和 AI 讨论架构
你："我要做一个 [项目描述]，
    预计有 [功能列表]，
    请建议整体架构"

AI：提供架构方案

你：理解、评估、调整

第 2 步：确定技术栈
你："基于这个架构，
    推荐什么技术栈？"

AI：提供技术选型建议

你：根据团队情况决策

第 3 步：设计模块划分
你："如何划分模块？
    模块间如何通信？"

AI：提供模块设计

你：审查、调整

第 4 步：定义接口
你："定义各模块的接口"

AI：生成接口定义

你：审查、确认

第 5 步：逐个实现模块
用 Vibe Coding 实现每个模块
```

#### 架构决策记录

```
创建 ARCHITECTURE.md：

# 架构设计

[架构图或描述]

## 关键决策
1. 为什么用 PostgreSQL？
   - 需要关系型数据

   - Session 存储
   - 缓存热点数据

## 数据流
[数据流图或描述]

[API 接口文档]
```

## 4. 不同 AI 工具的组合使用

不同 AI 工具有不同优势，组合使用能发挥最大价值。

### 4.1 Claude、ChatGPT、Copilot 的优势场景

#### Claude 的优势

```
最适合：
复杂的技术讨论
需要深入理解的场景
学习新技术
架构设计讨论
代码审查和优化建议

特点：
- 理解能力强
- 解释清晰
- 适合对话式学习
- 代码质量高

使用场景：
"我想理解 React 的渲染机制"
"帮我设计这个系统的架构"
"审查这段代码，给出改进建议"
```

#### ChatGPT 的优势

```
最适合：
快速查询
简单问题
代码片段生成
文档编写
多语言支持

特点：
- 响应快
- 知识面广
- 插件生态丰富
- 多模态能力

使用场景：
"这个错误是什么意思？"
"写一个正则表达式匹配邮箱"
"生成 API 文档"
```

#### GitHub Copilot 的优势

```
最适合：
代码补全
重复性代码
标准模式代码
日常编码
IDE 内使用

特点：
- 无缝集成
- 实时补全
- 上下文理解好
- 不打断流程

使用场景：
- 写函数时自动补全
- 生成测试用例
- 重复性代码
```

### 4.2 工具切换策略

#### 策略 1：按阶段切换

```
探索阶段：Claude
- 讨论技术方案
- 学习新概念
- 架构设计

实现阶段：Copilot
- 日常编码
- 代码补全
- 快速实现

问题解决：ChatGPT
- 快速查询错误
- 查找解决方案
- 搜索文档

审查阶段：Claude
- 代码审查
- 优化建议
- 重构方案
```

#### 策略 2：按任务类型切换

```
复杂任务：Claude
- 需要深入讨论
- 需要理解业务
- 需要设计方案

简单任务：Copilot
- 标准化代码
- 重复性工作
- 快速补全

查询任务：ChatGPT
- 快速查询
- 简单问题
- 文档查找
```

#### 策略 3：组合使用

```
例子：实现新功能

第 1 步：用 Claude 讨论方案
"我想实现 [功能]，
有哪些方案？
推荐哪个？"

第 2 步：用 Copilot 实现
在 IDE 中编码
Copilot 自动补全

第 3 步：遇到问题用 ChatGPT
"这个错误怎么解决？"
快速查询

第 4 步：用 Claude 审查
"审查这段代码，
有什么问题？"

结果：发挥各工具优势
```

```
   ├─ 是 → Claude
   └─ 否 → 根据具体情况
```

## 5. 元认知与自我优化

最高级的策略是**持续改进自己的 Vibe Coding 能力**。

### 5.1 记录和分析自己的 Vibe Coding 过程

#### 记录什么

```
每次 Vibe Coding 后记录：

1. 任务描述
   - 要做什么
   - 预期时间
   - 实际时间

2. 使用的方法
   - 用了哪些技巧
   - 哪些有效
   - 哪些无效

3. 遇到的问题
   - 什么问题
   - 如何解决
   - 花了多久

4. 产出质量
   - 代码质量如何
   - 是否符合预期
   - 需要返工吗

5. 经验教训
   - 学到了什么
   - 下次如何改进
```

#### 记录方式

```
方式 1：简单日志
2024-01-15
任务：实现用户登录
时间：2 小时（预期 1 小时）
方法：两步法（先讨论方案，再实现）
问题：密码加密不熟悉，花了 30 分钟学习
质量：良好，代码清晰
教训：下次先学习不熟悉的技术

方式 2：结构化记录
创建 LEARNING.md：
# Vibe Coding 学习日志

## 2024-01-15 用户登录
- 任务：[描述]
- 方法：[使用的技巧]
- 问题：[遇到的问题]
- 解决：[如何解决]
- 教训：[经验教训]

方式 3：使用工具
- Notion
- Obsidian
- 简单的 Markdown 文件
```

#### 定期回顾

```
每周回顾：
□ 这周做了什么？
□ 哪些做得好？
□ 哪些需要改进？
□ 学到了什么？

每月回顾：
□ 这个月的进步？
□ 效率提升了吗？
□ 代码质量如何？
□ 新掌握的技巧？

每季度回顾：
□ 整体能力提升？
□ 还有哪些不足？
□ 下季度目标？
```

### 5.2 识别个人模式

#### 识别有效模式

```
通过记录，识别你的有效模式：

"我发现的模式：
1. 用两步法（先讨论再实现）效率最高
2. 小步迭代比大步修改更可控
3. 早上精力好，适合做复杂任务
4. 下午适合做重复性工作
5. 每 1 小时休息一次效率更高"

这些是你的个人模式
```

#### 识别无效模式

```
识别你的反模式：

"我发现的问题：
1. 直接要代码（不讨论方案）经常返工
2. 一次改太多地方容易出错
3. 不测试就继续经常积累问题
4. 晚上写代码质量差
5. 连续工作超过 2 小时效率下降"

避免这些反模式
```

#### 识别个人偏好

```
了解你的偏好：

"我的偏好：
1. 喜欢用 Claude（解释清晰）
2. 喜欢简单的技术栈
3. 喜欢看到快速反馈
4. 不喜欢过度设计
5. 重视代码可读性"

根据偏好调整工作方式
```

### 5.3 持续改进

#### 改进循环

```
PDCA 循环：

Plan（计划）：
- 设定改进目标
- 制定改进计划

Do（执行）：
- 尝试新方法
- 记录过程

Check（检查）：
- 评估效果
- 对比之前

Act（行动）：
- 保留有效的
- 调整无效的
- 继续改进
```

#### 改进方向

```
方向 1：提升效率
- 减少返工
- 提高准确性
- 加快速度

方向 2：提升质量
- 代码更清晰
- 更少 bug
- 更好的设计

方向 3：扩展能力
- 学习新技术
- 掌握新工具
- 尝试新方法

方向 4：优化流程
- 简化步骤
- 自动化重复工作
- 建立模板
```

#### 实践建议

```
建议 1：设定小目标
不要：一次改进所有
而是：每次改进一个方面

例子：
本周目标：提升需求描述能力
- 使用需求模板
- 每次都明确约束
- 记录效果

建议 2：实验验证
不确定某个方法是否有效？
- 尝试一周
- 记录数据
- 对比效果
- 决定是否采用

建议 3：向他人学习
- 看优秀项目的代码
- 学习他人的方法
- 参考最佳实践
- 吸收有用的

建议 4：分享和教学
- 写博客记录经验
- 教别人 Vibe Coding
- 在教学中深化理解
- 获得反馈改进
```

#### 成长路径

```
初学者（0-3 个月）：
- 掌握基本工作流
- 能完成简单项目
- 建立基本直觉

熟练者（3-12 个月）：
- 掌握进阶技巧
- 能处理复杂项目
- 有自己的模式库

专家（12 个月+）：
- 精通各种策略
- 能指导他人
- 持续创新方法
```
